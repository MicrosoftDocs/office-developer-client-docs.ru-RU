# <a name="adrlist"></a>ADRLIST

**Относится к**: Outlook 2013 | Outlook 2016 
  
Описывает ноль или более свойств, относящихся к одному или нескольким получателям. 
  
|||
|:-----|:-----|
|Файл заголовка:  <br/> |MAPIDEFS. h  <br/> |
|Связанные макросы:  <br/> |[Кбадрлист](cbadrlist.md), [кбневадрлист](cbnewadrlist.md), [кбневадрлист](cbnewadrlist.md) <br/> |
   
```cpp
typedef struct _ADRLIST
{
  ULONG cEntries;
  ADRENTRY aEntries[MAPI_DIM];
} ADRLIST, FAR *LPADRLIST;

```

## <a name="members"></a>Members

**Центриес**
  
> Количество записей в массиве, указанном элементом **аентриес** . 
    
**Аентриес**
  
> Массив структур [адрентри](adrentry.md) , одна структура для каждого получателя. 
    
## <a name="remarks"></a>Примечания

Структура **ADRLIST** содержит одну или несколько структур **адрентри** , каждый из которых описывает свойства получателя. Получатель может быть неразрешимым. Это означает, что в массиве значений свойств отсутствует идентификатор записи. Разрешенный получатель означает, что включено свойство (код **\_EntryID** ([PidTagEntryId](pidtagentryid-canonical-property.md))). Как правило, у разрешенных получателей также есть адрес электронной почты для свойства **пр_емаил_аддресс** ([PidTagEmailAddress](pidtagemailaddress-canonical-property.md)). Однако адрес электронной почты не является обязательным. Структуры **ADRLIST** используются, например, для описания списка получателей исходящих сообщений и MAPI для отображения записей в адресной книге. 
  
Структуры **ADRLIST** похожи на структуры [SRowSet](srowset.md) структуры, используемые для представления строк в таблицах. На самом деле эти две структуры разрабатываются так, чтобы их можно было использовать в качестве взаимозаменяемых. Оба содержат массив структур, описывающих группу свойств, и число значений в массиве. В отличие от структуры **ADRLIST** , массив содержит структуры [адрентри](adrentry.md) , в структуре **SRowSet** массив содержит [сров](srow.md) структуры. Структуры **адрентри** и структуры **сров** идентичны в макете. Так как структуры **ADRLIST** и **SRowSet** следуют одним и тем же правилам выделения, то структура **SRowSet** , полученная из таблицы содержимого контейнера адресной книги, может быть приведена к структуре **ADRLIST** и использоваться как есть. 
  
На следующем рисунке показан макет структуры **ADRLIST** . 
  
**Компоненты ADRLIST**
  
![Компоненты ADRLIST] (media/amapi_18.gif "Компоненты ADRLIST")
  
Части **адрентри** и [Спропвалуе](spropvalue.md) в структуре **ADRLIST** должны быть распределены и освобождены независимо от других частей. То есть каждая структура **спропвалуе** должна выделяться отдельно после того, как память для структуры **адрентри** была выделена и освобождена до высвобождения структуры **адрентри** . Такая независимость при обработке памяти позволяет получателям и свойствам получателей свободно добавляться или удаляться из списка адресов. 
  
Для выделения и освобождения структуры **ADRLIST** и всех ее частей необходимо использовать функции [мапиаллокатебуффер](mapiallocatebuffer.md) и [мапифрибуффер](mapifreebuffer.md) . 
  
Если список получателей слишком большой и не помещается в память, клиенты могут вызвать метод [iMessage:: модифиреЦипиентс](imessage-modifyrecipients.md) для работы с подмножеством списка. В этой ситуации клиенты не должны использовать общие диалоговые окна адресной книги. 
  
Дополнительные сведения о том, как выделить память для структур **адрентри** , можно узнать в статье [Управление памятью для структур ADRLIST и SRowSet](managing-memory-for-adrlist-and-srowset-structures.md).
