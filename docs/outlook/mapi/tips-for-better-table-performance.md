# <a name="tips-for-better-table-performance"></a>Советы по повышению производительности таблиц
  
**Область применения**: Outlook 2013 | Outlook 2016 
  
Поскольку многие операции таблицы могут отнимать много времени и нет способа указать прогресс, полезно использовать следующие методы для повышения производительности:
  
- **Make [IMAPITable : IUnknown](imapitableiunknown.md) calls in the correct order**
    
   Клиенты и поставщики услуг могут работать со столами различными способами. Они могут открыть таблицу и получить данные для всех строк с помощью набора столбцов по умолчанию и сортировки. Кроме того, они могут изменить это представление таблицы по умолчанию, изменив набор столбцов, изменив порядок сортировки или настроив ограничение, чтобы сузить область таблицы. Пользователи таблиц, которые намерены выполнить одну или несколько из этих операций до получения данных, должны выполнять их в следующем порядке:
    
    1. Определение столбца с [IMAPITable::SetColumns](imapitable-setcolumns.md).
        
    2. Установить ограничение с [помощью IMAPITable::Restrict](imapitable-restrict.md).
        
    3. Определение порядка сортировки [с помощью IMAPITable::SortTable](imapitable-sorttable.md).
    
    Выполнение этих задач в этом порядке ограничивает количество строк и столбцов, которые будут сортироваться, тем самым повышая производительность.
    
- **Задержка операции с помощью флага TBL_BATCH, если это возможно**
    
    Установка флага ПАКЕТ TBL на методе позволяет реализации таблицы собирать несколько вызовов, прежде чем действовать \_ на любом из них. Вместо того, чтобы делать потенциально много вызовов на удаленный сервер; Реализации таблицы может сделать один, выполняя все операции одновременно. Результаты операций не оцениваются до тех пор, пока они не будут необходимы. 
    
    Например, если клиент вызывает [IMAPITable:::Restrict,](imapitable-restrict.md) чтобы указать ограничение с набором флага ПАКЕТ TBL, ограничение не вступает в силу, пока клиент не вызывает \_ [IMAPITable::QueryRows](imapitable-queryrows.md) для получения данных. Это позволяет реализации таблицы объединить работу двух вызовов в один. Пользователи таблиц, которые пользуются флагом TBL BATCH, должны знать, что обработка ошибок в этих условиях может \_ быть более сложной. 
    
    Поскольку обработка ошибок отложенной операции аналогична обработке ошибок при зачете флага MAPI DEFERRED_ERRORS, см. в дополнительных сведениях об отложении ошибок \_ [MAPI.](deferring-mapi-errors.md) 
    
- **Сохранение кэша часто используемых свойств**
    
    Поставщики услуг, реализующие таблицы, могут уменьшить время, необходимое для создания представления путем кэшинга копий часто используемых свойств объектов. Сохранение копии этих свойств в памяти позволяет сохранить их при каждом восстановлении представления.
