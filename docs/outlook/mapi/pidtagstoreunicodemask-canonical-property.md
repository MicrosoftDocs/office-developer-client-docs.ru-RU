# <a name="pidtagstoreunicodemask-canonical-property"></a>Каноническое свойство PidTagStoreUnicodeMask

  
  
**Относится к**: Outlook 2013 | Outlook 2016 
  
Содержит битовуюmass флагов, которые клиентские приложения должны запросить, чтобы определить характеристики хранения сообщений.
  
|||
|:-----|:-----|
|Связанные свойства:  <br/> |PR_STORE_UNICODE_MASK  <br/> |
|Идентификатор:  <br/> |0x340F  <br/> |
|Тип данных:  <br/> |PT_LONG  <br/> |
|Область:  <br/> |Хранилище сообщений MAPI  <br/> |
   
## <a name="remarks"></a>Примечания

Это свойство раскрывает возможности хранения сообщений для клиентских приложений, планируя отправку сообщения. Флаги могут упростить принятие решений клиентом или другим хранилищем, например, нужно ли отправлять PR_BODY **(** [PidTagBody)](pidtagbody-canonical-property.md)или только PR_RTF_COMPRESSED **(** [PidTagRtfCompressed).](pidtagrtfcompressed-canonical-property.md) Клиент никогда не должен устанавливать это свойство. Попытка возвращает **MAPI_E_COMPUTED**. 
  
Для этого свойства можно установить один или несколько из следующих флагов: 
  
STORE_ANSI_OK
  
> (131072, 0x00020000) Хранилище сообщений поддерживает свойства, содержащие символы Американского национального института стандартов (ANSI) (8-битные).
    
STORE_ATTACH_OK 
  
> (32, 0x00000020) Хранилище сообщений поддерживает привязку и вложения объектов (OLE) или вложения, не вложенные в OLE, в сообщения. 
    
STORE_CATEGORIZE_OK 
  
> (1024, 0x00000400) Хранилище сообщений поддерживает классификацию представлений таблиц. 
    
STORE_CREATE_OK 
  
> (16, 0x00000010) Хранилище сообщений поддерживает создание новых сообщений. 
    
STORE_ENTRYID_UNIQUE 
  
> (1, 0x00000001) Идентификаторы записей для объектов в хранилище сообщений уникальны, то есть никогда не повторно в течение всего жизненного года хранения. 
    
STORE_HTML_OK 
  
> (65536, 0x00010000) Хранилище сообщений поддерживает HTML-сообщения, хранимые в свойстве **PR_BODY_HTML** ([PidTagBodyHtml).](pidtagbodyhtml-canonical-property.md) Обратите **внимание STORE_HTML_OK** что в версиях MAPIDEFS не заданная STORE_HTML_OK не определена. H, которые входят в состав Microsoft Exchange 2000 Server и более ранних. Если в среде разработки используется MAPIDEFS. H-файл, который не **включает** STORE_HTML_OK, используйте значение "0x00010000". 
    
STORE_ITEMPROC
  
> (2097152, 0x00200000) В упакованном хранилище PST-данных указывает, что когда новое сообщение поступает в магазин, хранилище обрабатывает правила и фильтрацию нежелательной почты отдельно. Магазин вызывает [IMAPISupport::Notify](imapisupport-notify.md), setting **fnevNewMail** в структуре [NOTIFICATION,](notification.md) которая передается в качестве параметра, а затем передает сведения о новом сообщении клиенту прослушивания. Впоследствии, когда прослушивающий клиент получает уведомление, к сообщению не применяются правила. 
    
STORE_LOCALSTORE
  
> (524288, 0x00080000) Этот флаг зарезервирован и не должен использоваться.
    
STORE_MODIFY_OK 
  
> (8, 0x00000008) Хранилище сообщений поддерживает изменение существующих сообщений. 
    
STORE_MV_PROPS_OK 
  
> (512, 0x00000200) Хранилище сообщений поддерживает многооценные свойства, гарантирует стабильность порядка значений в многоценном свойстве в ходе операции сохранения и поддерживает моментацию многоценных свойств в таблицах. 
    
STORE_NOTIFY_OK 
  
> (256, 0x00000100) Хранилище сообщений поддерживает уведомления. 
    
STORE_OLE_OK 
  
> (64, 0x00000040) Хранилище сообщений поддерживает вложения OLE. Данные OLE доступны через **интерфейс IStorage,** например через **свойство PR_ATTACH_DATA_OBJ** ([PidTagAttachDataObject).](pidtagattachdataobject-canonical-property.md) 
    
STORE_PUBLIC_FOLDERS 
  
> (16384, 0x00004000) Папки в этом хранилище являются общедоступными (несколькими пользователями), а не частными (возможно, с несколькими экземплярами, но не с несколькими пользователями). 
    
STORE_PUSHER_OK
  
> (8388608, 0x00800000) Обработщик протокола MAPI не будет обходить хранилище, и хранилище будет отправлять индексе все изменения с помощью уведомлений для индексации сообщений.
    
STORE_READONLY 
  
> (2, 0x00000002) Все интерфейсы для хранения сообщений имеют доступ только для чтения. 
    
STORE_RESTRICTION_OK 
  
> (4096, 0x00001000) Хранилище сообщений поддерживает ограничения. 
    
STORE_RTF_OK 
  
> (2048, 0x00000800) Хранилище сообщений поддерживает сообщения в формате RTF (обычно сжатые),  а само хранилище сохраняет PR_BODY и **PR_RTF_COMPRESSED** синхронизацию. 
    
STORE_SEARCH_OK 
  
> (4, 0x00000004) Хранилище сообщений поддерживает папки результатов поиска. 
    
STORE_SORT_OK 
  
> (8192, 0x00002000) Хранилище сообщений поддерживает сортировку представлений таблиц. 
    
STORE_SUBMIT_OK 
  
> (128, 0x00000080) Хранилище сообщений поддерживает пометку сообщения для отправки. 
    
STORE_UNCOMPRESSED_RTF 
  
> (32768, 0x00008000) Хранилище сообщений поддерживает хранение измененных текстовых сообщений формы (RTF) в несжимаемой форме. Несхваченный поток RTF идентифицирован значением **dwMagicUncompressedRTF** в загоне потока. Значение **dwMagicUncompressedRTF** определяется в RTFLIB. H-файл. 
    
STORE_UNICODE_OK
  
> (262144, 0x00040000) Хранилище сообщений поддерживает свойства, содержащие символы Юникода.
    
Версию сообщения в RTF всегда можно сохранить, даже если хранилище сообщений не имеет RTF-данных. Если STORE_RTF_OK для определенного магазина не заданной, клиент с версиями RTF должен сам вызывать функцию [RTFSync,](rtfsync.md) чтобы синхронизировать версии **PR_BODY** и **PR_RTF_COMPRESSED** для текстового содержимого. RTF всегда хранится в **PR_RTF_COMPRESSED** независимо от того, сжат ли он на самом деле.
