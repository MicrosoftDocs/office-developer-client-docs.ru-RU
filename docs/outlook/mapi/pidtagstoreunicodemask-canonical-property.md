# <a name="pidtagstoreunicodemask-canonical-property"></a>Каноническое свойство PidTagStoreUnicodeMask

  
  
**Применимо к**: Outlook 2013 | Outlook 2016 
  
Содержит битовую маску флагов, клиентские приложения необходимо запросить для определения характеристик хранилища сообщений.
  
|||
|:-----|:-----|
|Связанные свойства:  <br/> |PR_STORE_UNICODE_MASK  <br/> |
|Идентификатор:  <br/> |0x340F  <br/> |
|Тип данных:  <br/> |PT_LONG  <br/> |
|Область:  <br/> |Хранение сообщений MAPI  <br/> |
   
## <a name="remarks"></a>Замечания

Это свойство, включающие описание возможностей хранилища сообщений на клиентские приложения планирования отправить сообщение. Флаги могут быть упрощены решения с клиента или другого хранилища, например необходимость отправки **PR_BODY** ([PidTagBody](pidtagbody-canonical-property.md)) или только **PR_RTF_COMPRESSED** ([PidTagRtfCompressed](pidtagrtfcompressed-canonical-property.md)). Клиент никогда не следует устанавливать это свойство. Попытка возвращает **MAPI_E_COMPUTED**. 
  
Для этого свойства может быть установлено одно или несколько из следующих флагов: 
  
STORE_ANSI_OK
  
> (131072, 0x00020000) Хранилище сообщений поддерживает свойства символами (8 бит) American National стандартов института (ANSI).
    
STORE_ATTACH_OK 
  
> (32, 0x00000020) Хранилище сообщений поддерживает Object Linking и внедрения (OLE) или не OLE вложений в сообщения. 
    
STORE_CATEGORIZE_OK 
  
> (1024, 0x00000400) Хранилище сообщений поддерживает по категориям представления таблицы. 
    
STORE_CREATE_OK 
  
> (16, 0x00000010) Хранилище сообщений поддерживает создание новых сообщений. 
    
STORE_ENTRYID_UNIQUE 
  
> (1, 0x00000001) Идентификаторы записей для объектов в хранилище сообщений являются уникальными, то есть, никогда не повторно использовать на протяжении хранилища. 
    
STORE_HTML_OK 
  
> (65536, 0x00010000) Хранилище сообщений поддерживает HTML-сообщений, хранящиеся в свойстве **PR_BODY_HTML** ([PidTagBodyHtml](pidtagbodyhtml-canonical-property.md)). Обратите внимание на то, что **STORE_HTML_OK** не определен в версиях MAPIDEFS. H, которые включены в Microsoft Exchange 2000 Server и более ранних версий. Если в среде разработки используются MAPIDEFS. H-файл, не включают **STORE_HTML_OK**, вместо этого используйте значение «0x00010000». 
    
STORE_ITEMPROC
  
> (2097152, 0x00200000) В банке оболочку PST-файлов указывает, что при поступлении нового сообщения в хранилище, store работает правил и нежелательной почты фильтрации отдельно обработки в окне сообщения. Хранилище вызовы [IMAPISupport::Notify](imapisupport-notify.md)параметр **fnevNewMail** в структуре [УВЕДОМЛЕНИЙ](notification.md) , который передается как параметр, а затем передает сведения о новых сообщений для прослушивания клиента. Следовательно когда прослушивания клиент получает уведомление, не будет обрабатывать правил для сообщения. 
    
STORE_LOCALSTORE
  
> (524288, 0x00080000) Этот флаг зарезервирован и не должен использоваться.
    
STORE_MODIFY_OK 
  
> (8, 0x00000008) Хранилище сообщений поддерживает изменения существующего сообщения. 
    
STORE_MV_PROPS_OK 
  
> (512, 0x00000200) Хранилище сообщений поддерживает многозначные свойства, гарантирует стабильность значение заказ в свойством во всем сохранения операции и поддерживает создание многозначных свойств в таблицах. 
    
STORE_NOTIFY_OK 
  
> (256, 0x00000100) Хранилище сообщений поддерживает уведомления. 
    
STORE_OLE_OK 
  
> (64, 0x00000040) Хранилище сообщений поддерживает вложения OLE. Данные OLE доступен через интерфейс **IStorage** , например, то доступен через свойство **PR_ATTACH_DATA_OBJ** ([PidTagAttachDataObject](pidtagattachdataobject-canonical-property.md)). 
    
STORE_PUBLIC_FOLDERS 
  
> (16384, 0x00004000) Папки в этом хранилище — общедоступный (многопользовательский) не закрытый (возможно нескольких экземпляров, но не нескольких пользователей). 
    
STORE_PUSHER_OK
  
> (8388608, 0x00800000) Обработчик протокола MAPI не будет выполнять обход контента хранилище и хранилище отвечает отказываться все изменения через уведомления для компонента индексирования индексируются сообщений.
    
STORE_READONLY 
  
> (2, 0x00000002) Все интерфейсы для хранилища сообщений с уровнем доступа только для чтения. 
    
STORE_RESTRICTION_OK 
  
> (4096, 0x00001000) Хранилище сообщений поддерживает ограничения. 
    
STORE_RTF_OK 
  
> (2048, 0x00000800) Хранилище сообщений поддерживает форматированный текст (RTF) сообщения, обычно сжатием и само хранилище отслеживает **PR_BODY** и **PR_RTF_COMPRESSED** синхронизации. 
    
STORE_SEARCH_OK 
  
> (4, 0x00000004) Хранилище сообщений поддерживает папки результатов поиска. 
    
STORE_SORT_OK 
  
> (8192, 0x00002000) Хранилище сообщений поддерживает сортировки представления таблицы. 
    
STORE_SUBMIT_OK 
  
> (128, 0x00000080) Хранилище сообщений поддерживает пометки сообщения для отправки. 
    
STORE_UNCOMPRESSED_RTF 
  
> (32768, 0x00008000) Хранилище сообщений поддерживает хранение сообщений текста (RTF) revisable формы в несжатую форму. Поток несжатую RTF определяется значение **dwMagicUncompressedRTF** в заголовке потока. Значение **dwMagicUncompressedRTF** определен в RTFLIB. H-файл. 
    
STORE_UNICODE_OK
  
> (262144, 0x00040000) Хранилище сообщений поддерживает свойства, содержащее знаки Юникода.
    
Более RTF версии сообщения могут всегда храниться, даже в том случае, если хранилище сообщений не поддерживающими формат RTF. Если бит STORE_RTF_OK не установлен для конкретного хранилища, обслуживание RTF версий клиента самого вызовите функцию [RTFSync](rtfsync.md) сохранение версий **PR_BODY** и **PR_RTF_COMPRESSED** , синхронизированы для текстовое содержимое. Формат RTF всегда сохраняется в **PR_RTF_COMPRESSED**ли фактически сжимается или нет.
