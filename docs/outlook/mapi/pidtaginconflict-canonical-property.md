# <a name="pidtaginconflict-canonical-property"></a>Каноническое свойство PidTagInConflict

  
  
**Область применения**: Outlook 2013 | Outlook 2016 
  
Содержит значение TRUE, если вложение представляет альтернативную реплику.
  
|||
|:-----|:-----|
|Связанные свойства:  <br/> |ПР_ИН_КОНФЛИКТ  <br/> |
|Идентификатор:  <br/> |0x666C  <br/> |
|Тип данных:  <br/> |PT_BOOLEAN  <br/> |
|Область:  <br/> |Заметка о конфликте  <br/> |
   
## <a name="remarks"></a>Примечания

Клиент и сервер электронной почты должны создать сообщение об устранении конфликта при обнаружении конфликта с текущей версией сообщения в реплике во время синхронизации. Важно понимать, что текущая версия сообщения в локальной реплике передается во время текущей операции синхронизации. Это происходит, когда конфликт уже существует на сервере, прежде чем все конфликтующие сообщения будут загружены в локальную реплику. Сообщение о конфликте при разрешении конфликтов должно быть синхронизировано как независимые реплики с конфликтующими Пклс. Само сообщение с разрешением конфликтов не должно синхронизироваться между клиентом и сервером; только независимые реплики должны быть передаваться в Exchange. После этого партнер синхронизации должен создать новое сообщение, которое соответствует структуре конфликтующего сообщения. Поэтому важно, чтобы клиент и сервер использовали один и тот же алгоритм для определения элемента "победителя". Для определения "победителя" должны быть применены следующие правила:
  
1. Время последнего изменения.
    
2. Более высокий идентификатор GUID CN (с использованием сравнения памяти) для прерывания связи.
