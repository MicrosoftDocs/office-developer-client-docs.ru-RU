# <a name="pidtaginconflict-canonical-property"></a>Каноническое свойство PidTagInConflict

  
  
**Относится к**: Outlook 2013 | Outlook 2016 
  
Содержит true, если вложение представляет альтернативную реплику.
  
|||
|:-----|:-----|
|Связанные свойства:  <br/> |PR_IN_CONFLICT  <br/> |
|Идентификатор:  <br/> |0x666C  <br/> |
|Тип данных:  <br/> |PT_BOOLEAN  <br/> |
|Область:  <br/> |Примечание о конфликте  <br/> |
   
## <a name="remarks"></a>Примечания

Почтовый клиент и сервер должны создать сообщение о конфликте при обнаружении конфликта с текущей версией сообщения в реплике во время синхронизации. Важно понимать, что текущая версия сообщения в локальной реплике была передана во время текущей операции синхронизации. Это произойдет, когда конфликт уже существует на сервере, прежде чем любое конфликтующие сообщения будут загружены в локализованную реплику. Сообщение с разрешением конфликта должно быть синхронизировано как независимая реплика с конфликтующие PCLs. Само сообщение о конфликте не должно синхронизироваться между клиентом и сервером; Следует обмениваться только независимыми репликами. Затем партнер по синхронизации должен создать новое сообщение, которое соответствует структуре сообщения о конфликте. Поэтому важно, чтобы клиент и сервер использовали один и тот же алгоритм для обнаружения элемента "заметь". Для обнаружения "хим" необходимо применить следующие правила:
  
1. Время последнего изменения.
    
2. Более высокий GUID CN (с помощью сравнения памяти) для разрыва связи.
