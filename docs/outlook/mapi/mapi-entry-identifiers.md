# <a name="mapi-entry-identifiers"></a>Идентификаторы записей MAPI

  
  
**Область применения**: Outlook 2013 | Outlook 2016 
  
Идентификаторы входа — это части двоичных данных, хранимые в структуре [ENTRYID,](entryid.md) которые используются для уникальной идентификации и открытия объекта MAPI. Большинство объектов MAPI имеют идентификаторы входа. Идентификаторы входа для объектов аналогичны именам файлов. Однако они не передаются и не могут использоваться в системах, помимо системы, на основе системы, в которая они были зародились. 
  
## <a name="entry-identifiers"></a>Идентификаторы записи

Поставщики хранилища сообщений назначают идентификаторы записей в хранилищах сообщений, папках и сообщениях; Поставщики адресных книг назначают их для адресных контейнеров книг, списков рассылки и пользователей сообщений. Идентификаторы записи также используются для открытия объекта, представленного строкой в таблице, например объекта состояния в таблице состояния. Объекты хранят идентификаторы записей в **свойстве PR_ENTRYID** [(PidTagEntryId).](pidtagentryid-canonical-property.md) 
  
Если поставщики услуг создают, назначают и изучают идентификаторы записей, клиентские приложения используют их только в качестве средств для открытия объектов. Для клиентов идентификаторы входа являются непрозрачными частями двоичных данных и не имеют ничего общего с системой обмена сообщениями. 
  
Клиенты звонят по методу [IMAPIProp::GetProps](imapiprop-getprops.md) объекта для получения свойства **PR_ENTRYID** или метода [IMAPITable::QueryColumns](imapitable-querycolumns.md) для  получения столбца, на который PR_ENTRYID свойства. 
  
Идентификаторы записи передаются в качестве параметров методам **OpenEntry** и **CompareEntryIDs.** Несколько объектов MAPI реализуют методы **OpenEntry** и **CompareEntryIDs.** С **помощью OpenEntry** клиенты могут открыть объект. С **помощью compareEntryID клиенты** могут сравнить два идентификатора записи, чтобы определить, относятся ли они к одному объекту. Так как идентификаторы записей не обязательно являются двоичными, клиенты должны сравнивать их с помощью метода **CompareEntryIDs.** 
  
Клиенты всегда должны передавать поставщику услуг идентификаторы записей с естественным соответствием, так как, хотя поставщики услуг должны обрабатывать идентификаторы записей, которые произвольно выровнены, это не всегда так. Естественно выровненный адрес памяти позволяет компьютеру получать доступ к любому типу данных, который он поддерживает по этому адресу, без создания ошибки выравнивания. Естественный коэффициент выравнивания обычно является одним и тем же фактором выравнивания, используемым в системе, и обычно составляет 8 bytes.
  
Идентификаторы записи приходят в двух типах: краткосрочный и долгосрочный. Краткосрочные идентификаторы записей быстрее создавать, но их уникальность гарантируется только в течение текущего сеанса на текущей рабочей станции. Долгосрочные идентификаторы записей имеют более длительный срок службы. Краткосрочные идентификаторы записей используются в основном для строк в таблицах и записей в диалоговых окнах, в то время как долгосрочные идентификаторы записей используются для многих объектов, таких как сообщения, папки и списки рассылки.
