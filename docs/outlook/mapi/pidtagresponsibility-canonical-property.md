# <a name="pidtagresponsibility-canonical-property"></a>Каноническое свойство PidTagResponsibility

  
  
**Область применения**: Outlook 2013 | Outlook 2016 
  
Содержит значение TRUE, если некоторые поставщики транспорта уже приняли ответственность за доставку сообщения этому получателю, и FALSE, если диспетчер очереди MAPI считает, что этот поставщик транспорта должен принимать ответственность.
  
|||
|:-----|:-----|
|Связанные свойства:  <br/> |ПР_РЕСПОНСИБИЛИТИ  <br/> |
|Идентификатор:  <br/> |0x0E0F  <br/> |
|Тип данных:  <br/> |PT_BOOLEAN  <br/> |
|Область:  <br/> |Несъемный MAPI  <br/> |
   
## <a name="remarks"></a>Комментарии

Когда диспетчер очереди MAPI представляет исходящее сообщение для поставщика транспорта с помощью [иксплогон:: субмитмессаже](ixplogon-submitmessage.md), этому свойству ПРИСВАИВАЕТСЯ значение false для всех получателей, для которых диспетчер очереди MAPI считает его ответственным, и true для всех другие получатели. Поставщик транспорта должен попытаться обрабатывать всех получателей, у которых для **пр_респонсибилити** ЗАДАНО значение false. После успешной отправки или неудачной ошибки отправки получателю, поставщик транспорта должен задать для этого свойства значение TRUE в исходном сообщении, чтобы указать, что он принял ответственность за этого получателя. 
  
Если после проверки получателя поставщик транспорта решает, что он не может или не должен обрабатывать его, то поставщик транспорта должен оставить для параметра **пр_респонсибилити** значение false. После этого диспетчер очереди MAPI будет искать другого поставщика транспорта, который может обработать этого получателя. В конечном счете, диспетчер очереди MAPI создает отчет о недоставке для всех получателей, для которых никакой поставщик транспорта не принимает ответственность. 
  
Если поставщик транспорта пытается и не может доставить сообщение, он должен вызвать метод [имаписуппорт:: статусреЦипс](imapisupport-statusrecips.md) , чтобы указать MAPI причины сбоя, чтобы MAPI мог создать отчет о недоставке.
