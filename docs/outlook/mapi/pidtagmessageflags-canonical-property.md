# <a name="pidtagmessageflags-canonical-property"></a>Каноническое свойство PidTagMessageFlags

  
  
**Относится к**: Outlook 2013 | Outlook 2016 
  
Содержит битовуюmass флагов, которые указывают на происхождение и текущее состояние сообщения. 
  
|||
|:-----|:-----|
|Связанные свойства:  <br/> |PR_MESSAGE_FLAGS  <br/> |
|Идентификатор:  <br/> |0x0E07  <br/> |
|Тип данных:  <br/> |PT_LONG  <br/> |
|Область:  <br/> |Общие сообщения  <br/> |
   
## <a name="remarks"></a>Примечания

Это свойство является непередаваемым свойством сообщения, которое вы получаете как на отправляющих, так и на принимающих концах передачи, с различными значениями в зависимости от клиентского приложения или поставщика магазина. Это свойство инициализируются клиентом или поставщиком хранилищем сообщений, когда сообщение создается и сохраняется в первый раз, а затем периодически обновляется поставщиком store сообщений, поставщиком транспорта и пулером MAPI по мере обработки сообщения и изменения его состояния. 
  
Это свойство существует в сообщении до и после отправки, а также во всех копиях полученного сообщения. Хотя оно не является свойством получателя, оно по-разному передается каждому получателю в зависимости от того, было ли оно прочитано или изменено этим получателем. 
  
Для этого свойства можно установить один или несколько из следующих флагов:
  
MSGFLAG_ASSOCIATED 
  
> Сообщение является связанным сообщением папки. Клиент или поставщик имеют доступ только для чтения к этому флагу. Флаг MSGFLAG_READ игнорируется для связанных сообщений, которые не сохраняют состояние чтения или непрочитанного сообщения. 
    
MSGFLAG_FROMME 
  
> Отправив сообщение, пользователь сообщения получил сообщение. Клиент или поставщик имеет доступ для чтения и записи к этому флагу до первого вызова [IMAPIProp::SaveChanges](imapiprop-savechanges.md) и после этого только для чтения. Этот флаг должен устанавливаться поставщиком транспорта. 
    
MSGFLAG_HASATTACH 
  
> Сообщение имеет по крайней мере одно вложение. Этот флаг соответствует свойству PR_HASATTACH **(** [PidTagHasAttachments).](pidtaghasattachments-canonical-property.md) Клиент имеет доступ только для чтения к этому флагу. 
    
MSGFLAG_NRN_PENDING 
  
> Для сообщения необходимо отправить непрочитанные отчеты. Клиент или поставщик имеют доступ только для чтения к этому флагу. 
    
MSGFLAG_ORIGIN_INTERNET 
  
> Входящие сообщения поступили через Интернет. Он исходил из-за пределов организации или из источника, который шлюз не может считать доверенным. Клиент должен отобразить соответствующее сообщение для пользователя. Поставщики транспорта устанавливают этот флаг; клиент имеет доступ только для чтения. 
    
MSGFLAG_ORIGIN_MISC_EXT 
  
> Входящие сообщения поступили по внешней ссылке, не относящуюся к X.400 или Интернету. Он исходил из-за пределов организации или из источника, который шлюз не может считать доверенным. Клиент должен отобразить соответствующее сообщение для пользователя. Поставщики транспорта устанавливают этот флаг; клиент имеет доступ только для чтения. 
    
MSGFLAG_ORIGIN_X400 
  
> Входящие сообщения поступили по ссылке X.400. Он исходил из-за пределов организации или из источника, который шлюз не может считать доверенным. Клиент должен отобразить соответствующее сообщение для пользователя. Поставщики транспорта устанавливают этот флаг; клиент имеет доступ только для чтения. 
    
MSGFLAG_READ 
  
> Сообщение помечено как прочитано. Это может произойти в результате вызова [IMessage::SetReadFlag](imessage-setreadflag.md) или [IMAPIFolder::SetReadFlags.](imapifolder-setreadflags.md) Клиенты также могут установить этот флаг, вызывая метод **IMAPIProp::SetProps** сообщения, прежде чем сообщение будет сохранено в первый раз. Этот флаг игнорируется, **если** MSGFLAG_ASSOCIATED установлен. 
    
MSGFLAG_RESEND 
  
> Сообщение включает запрос на операцию повторной работы с отчетом о ненастройстве. Клиент или поставщик имеет доступ для чтения и записи к этому флагу до первого вызова [IMAPIProp::SaveChanges](imapiprop-savechanges.md) и после этого только для чтения. 
    
MSGFLAG_RN_PENDING 
  
> Для сообщения необходимо отправить отчет о прочте.. Клиент или поставщик имеют доступ только для чтения к этому флагу. 
    
MSGFLAG_SUBMIT 
  
> Сообщение помечено для отправки в результате вызова [IMessage::SubmitMessage.](imessage-submitmessage.md) Поставщики store сообщений устанавливают этот флаг; клиент имеет доступ только для чтения. 
    
MSGFLAG_UNMODIFIED 
  
> Исходя извещение не было изменено с момента первого с момента его с момента с сохранения; входящие сообщения не были изменены с момента его доставки. 
    
MSGFLAG_UNSENT 
  
> Сообщение все еще находится в составлении. Он сохранен, но не отправлен. Клиент или поставщик имеет доступ для чтения и записи к этому флагу до первого вызова [IMAPIProp::SaveChanges](imapiprop-savechanges.md) и после этого только для чтения. Если клиент не устанавливает этот флажок к моменту отправки сообщения, поставщик магазина сообщений устанавливает его при запросе **IMessage::SubmitMessage.** Как правило, этот флаг очищается после того, как сообщение отправлено. 
    
Клиент или поставщик store сообщений может в любое время проверить текущее состояние сообщения, вызывая метод [IMAPIProp::GetProps,](imapiprop-getprops.md) чтобы прочитать значения флага. Клиент или поставщик также может вызвать метод [IMAPIProp::SetProps,](imapiprop-setprops.md) чтобы изменить любые флаги, которые в настоящее время имеют доступ на чтение и записи. 
  
Некоторые флаги всегда могут быть только для чтения. Некоторые из них читают и записывают до первого вызова метода [IMAPIProp::SaveChanges,](imapiprop-savechanges.md) а затем становятся только для чтения, если используется **IMAPIProp::SetProps.** Одно из этих MSGFLAG_READ можно изменить позже с помощью [IMessage::SetReadFlag](imessage-setreadflag.md) или [IMAPIFolder::SetReadFlags.](imapifolder-setreadflags.md) 
  
Начальные значения этого свойства обычно MSGFLAG_UNSENT и MSGFLAG_UNMODIFIED, чтобы указать сообщение, которое еще не было отправлено или изменено. Когда сообщение сохраняется во второй раз, поставщик хранилище сообщений очищает флаг MSGFLAG_UNMODIFIED сообщения. Еще одно значение, которое может установить поставщик для хранения сообщений при его MSGFLAG_HASATTACH флаге, указывающее на то, что сообщение имеет одно или несколько вложений. Свойство **PR_HASATTACH** вычисляется из этого параметра. 
  
Когда клиент вызывает метод [IMessage::SubmitMessage](imessage-submitmessage.md) для отправки сообщения, поставщик службы хранения сообщений создает его копию для пулера MAPI и обновляет это свойство, устанавливая флаг MSGFLAG_SUBMIT сообщения. Поставщик store сообщений также MSGFLAG_UNSENT, если он еще не установлен. MSGFLAG_SUBMIT, что **submitMessage** был вызван, начинает процесс отправки и что сообщение теперь только для чтения клиенту. MSGFLAG_UNSENT указывает, что пулер MAPI обработчик сообщений. Если процесс отправки отменен, поставщик store сообщений сбрасывает этот флаг. 
  
Когда сообщение отправляется поставщику транспорта для доставки, поставщик транспорта устанавливает флаг MSGFLAG_FROMME, если отправитель имеет ту же учетную запись на сервере обмена сообщениями, что и сообщение было получено. Поставщики транспорта MSGFLAG_FROMME для входящих сообщений, отправленных текущим пользователем. Клиент может использовать это значение, чтобы определить, что лучше показывать имена получателей в таблице содержимого папки "Отправленные", чем имена отправитель. Сообщения, сохраненные в процессе композиции и еще не отправленные, также должны отображаться с именами получателей, а не с именами отправитель. 
  
Для входящих сообщений поставщик хранилище сообщений очищает флаг MSGFLAG_READ, чтобы сбросить состояние чтения. Клиент может установить или очистить флаг MSGFLAG_READ, если необходимо изменить состояние чтения и управлять отправкой отчетов о прочтях и непрочитаных сообщениях, вызывая метод [IMessage::SetReadFlag сообщения](imessage-setreadflag.md) или метод [IMAPIFolder::SetReadFlags](imapifolder-setreadflags.md) его папки. Основное различие между этими методами, кроме реализующих их объектов, заключается в том, что метод папки может влиять на одно, несколько или все сообщения в папке. Метод сообщения влияет на одно сообщение. 
  
Клиент также должен проверить входящие сообщения на MSGFLAG_ORIGIN_X400, MSGFLAG_ORIGIN_INTERNET и MSGFLAG_ORIGIN_MISC_EXT сообщений. Эти флаги устанавливаются поставщиком входящий транспорта и указывают, что сообщение поступило из источника, который шлюз не может считать доверенным. Это означает, что сообщение исходит из-за пределов организации или из внутренней сети, но с рабочей станции, не известной шлюзу. В любом случае не может быть подтверждено удостоверение отправляющего, и существует риск внедрения в организацию компьютера с вирусом. Клиент должен отобразить пользователю предупреждающие сообщения и предложить удалить сообщение, не открывая его. 
  
Поставщики хранения сообщений устанавливают флаг MSGFLAG_UNMODIFIED для входящих сообщений. MSGFLAG_UNMODIFIED указывает, что сообщение не было изменено с момента доставки. Клиент не может очистить это значение после того, как оно было установлено поставщиком store сообщений.
