# <a name="upload-delete-status-state"></a>Состояние отправки состояния удаления

  
  
**Область применения**: Outlook 2013 | Outlook 2016 
  
 В этом разделе описывается, что происходит во время отправки состояния удаления для конечного автомата репликации. 
  
## <a name="quick-info"></a>Краткие сведения

|||
|:-----|:-----|
|Идентификатор состояния:  <br/> |**ЛР_СИНК_УПЛОАД_МЕССАЖЕ_ДЕЛ** <br/> |
|Связанная структура данных:  <br/> |**[UPDEL](updel.md)** <br/> |
|Из этого состояния:  <br/> |[Состояние отправки таблицы](upload-table-state.md) <br/> |
|В это состояние:  <br/> |Состояние отправки таблицы  <br/> |
   
> [!NOTE]
> Конечный автомат репликации — это детерминированный конечный автомат. Клиент, который отменяет от одного состояния к другому, должен, в конечном итоге, вернуться к предыдущему из последнего. 
  
## <a name="description"></a>Описание

Это состояние инициирует обновление на сервере, который содержит элементы Outlook (почта, календарь, контакт, задача, Примечание или журнал), которые были удалены в папке в локальном хранилище, указанном в предыдущем состоянии таблицы отправки. В этом состоянии Outlook инициализирует элементы в связанной структуре данных **упдел** информацией об элементах, которые были удалены или перемещены из папки. 
  
Затем клиент удаляет указанные элементы из папки на сервере. Чтобы отличать элементы, которые были перемещены в отличие от удаления, клиент должен проверить элементы *пупмов* , определенные в структуре **упдел** . 
  
При завершении этого состояния Outlook Очищает внутреннюю информацию, указывающую на то, что элемент был удален; Поэтому в Outlook больше не будет записей об элементе. Локальное хранилище возвращается в состояние таблицы отправки.
