# <a name="imsgstorestorelogoff"></a>IMsgStore::StoreLogoff

  
  
**Область применения**: Outlook 2013 | Outlook 2016 
  
Включает упорядоченный журнал магазина сообщений.
  
```cpp
HRESULT StoreLogoff(
  ULONG FAR * lpulFlags
);
```

## <a name="parameters"></a>Parameters

 _lpulFlags_
  
> [in, out] Битмашка флагов, которые контролируют журнал из магазина сообщений. На входе все флаги, заданные для этого параметра, являются взаимоисключающими; вызываемая должна указывать только один флаг на один вызов. При входе допустимы следующие флаги:
    
LOGOFF_ABORT 
  
> Все действия поставщика транспорта для этого магазина сообщений должны быть остановлены перед входом в систему. Управление возвращается вызываемой после того, как действие остановлено. Если какие-либо действия поставщика транспорта происходят, журнал не происходит, и не происходит никаких изменений в поведении пулера MAPI или поставщиков транспорта. Если деятельность поставщика транспорта простаивает, пульпер MAPI освобождает магазин. 
    
LOGOFF_NO_WAIT 
  
> Хранилище сообщений не должно ждать сообщений от поставщиков транспорта перед закрытием. Исходящие сообщения, готовые к отправлению, отправляются. Если этот магазин содержит почтовый ящик по умолчанию, все в процессе сообщения будут получены, а затем дополнительный прием отключен. Когда все действия завершены, пульпер MAPI освобождает магазин, и управление немедленно возвращается вызываемой. 
    
LOGOFF_ORDERLY 
  
> Хранилище сообщений не должно ждать информации от поставщиков транспорта перед закрытием. Сообщения, которые в настоящее время обрабатываются, завершались, но новые сообщения не обрабатываются. Когда все действия завершены, пульпер MAPI освобождает магазин, и управление немедленно возвращается поставщику магазина. 
    
LOGOFF_PURGE 
  
> Вход должен работать так же, как если бы флаг LOGOFF_NO_WAIT установлен, но для соответствующих поставщиков транспорта должен быть вызван метод [IXPLogon::FlushQueues](ixplogon-flushqueues.md) или [IMAPIStatus::FlushQueues.](imapistatus-flushqueues.md) Флаг LOGOFF_PURGE возвращает управление вызываемой после завершения. 
    
LOGOFF_QUIET 
  
> Если какие-либо действия поставщика транспорта происходят, журнал не должен происходить.
    
    The following flags are valid on output:
    
LOGOFF_INBOUND 
  
> В настоящее время прибывают входящие сообщения.
    
LOGOFF_OUTBOUND 
  
> Исходящие сообщения находятся в процессе отосла.
    
LOGOFF_OUTBOUND_QUEUE 
  
> Исходящие сообщения находятся в ожидании (то есть они находятся в исходящие сообщения).
    
## <a name="return-value"></a>Возвращаемое значение

S_OK 
  
> Журнал успешно завершен.
    
## <a name="remarks"></a>Примечания

Метод **IMsgStore::StoreLogoff** контролирует взаимодействие магазина сообщений и поставщиков транспорта во время процесса входа. Вызов **StoreLogoff** действителен только для магазинов сообщений, которые используются только вызываемой. Например, когда два клиента используют один и тот же хранилище сообщений и один из них вызывает **StoreLogoff,** магазин сообщений немедленно освобожден и управление возвращается вызываемому клиенту.
  
## <a name="notes-to-implementers"></a>Примечания для исполнителей

Сохраните флаги, которые передаются **в StoreLogoff,** и передайте их при вызове метода [IMAPISupport::StoreLogoffTransports.](imapisupport-storelogofftransports.md) Не **вызывайте StoreLogoffTransports** до тех пор, пока количество ссылок магазина сообщений не опустится до нуля. Несколько вызовов **в StoreLogoffTransports просто** переопишут сохраненные флаги. 
  
Если вызов в **StoreLogoff** не был выполнен до достижения нулевого отсчета в хранилище сообщений, установите флаг LOGOFF_ABORT в параметре _ulFlags,_ который передается в **StoreLogoffTransports.**
