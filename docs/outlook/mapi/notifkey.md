# <a name="notifkey"></a>NOTIFKEY

  
  
**Область применения**: Outlook 2013 | Outlook 2016 
  
Уникально определяет связь между раковиной рекомендации, источником консультаций и MAPI.
  
|||
|:-----|:-----|
|Файл заголовка:  <br/> |Mapispi.h  <br/> |
   
```cpp
typedef struct
{
  ULONG cb;
  BYTE ab[MAPI_DIM];
} NOTIFKEY, FAR *LPNOTIFKEY;

```

## <a name="members"></a>"Участники"

 **cb**
  
> Количество bytes в **члене ab.** 
    
 **ab**
  
> Массив bytes, описывающий ключ уведомления.
    
## <a name="remarks"></a>Примечания

Методы [подписки](imapisupport-subscribe.md) [и](imapisupport-notify.md) уведомления [IMAPISupport](imapisupportiunknown.md) используют структуру **NOTIFKEY** для создания уведомлений соответствующему источнику консультаций. 
  
Поставщики услуг создают ключи уведомлений при вызове метода **Advise** и хотят вызвать **Подписку** для обработки регистрации уведомлений и последующей отправки уведомлений. Ключом уведомления может быть идентификатор входа источника консультаций или любой другой идентифицирующий элемент, например константа. Например, поставщик магазина сообщений может использовать путь папки в качестве ключа уведомления. 
  
Ключ уведомления должен работать в нескольких процессах. 
  
Требования к области для ключа уведомлений напоминают требования к долгосрочному идентификатору входа. Однако, в отличие от идентификатора записи, ключ уведомления должен быть двоичным. Как правило, ключ уведомления включает значение **GUID,** определенное поставщиком услуг, за которым следуют другие сведения, характерные для поставщика, уникальные для объекта. 
  
Для обсуждения использования структуры **NOTIFKEY** для управления подключениями между раковинами-советами и объектами, которые генерируют уведомления, см. в сообщении о поддержке уведомлений о [событиях.](supporting-event-notification.md)
