# <a name="notifkey"></a>NOTIFKEY

  
  
**Относится к**: Outlook 2013 | Outlook 2016 
  
Уникально идентифицирует связь между осями консультации, источником консультаций и MAPI.
  
|||
|:-----|:-----|
|Файл заголовка:  <br/> |Mapispi.h  <br/> |
   
```cpp
typedef struct
{
  ULONG cb;
  BYTE ab[MAPI_DIM];
} NOTIFKEY, FAR *LPNOTIFKEY;

```

## <a name="members"></a>"Участники"

 **cb**
  
> Количествобайтов в **члене ab.** 
    
 **ab**
  
> Массив ветвей, описывающий ключ уведомления.
    
## <a name="remarks"></a>Примечания

Методы [Subscribe](imapisupport-subscribe.md) и [Notify](imapisupport-notify.md) [IMAPISupport](imapisupportiunknown.md) используют структуру **NOTIFKEY** для создания уведомлений соответствующему приемнику с уведомлением о соответствующем источнике консультаций. 
  
Поставщики услуг создают ключи уведомлений при вызове метода **Advise** и хотят вызвать **подписку** для обработки регистрации уведомлений и последующей отправки уведомлений. Ключ уведомления может быть идентификатором записи источника консультации или любым другим идентифицирующий элемент, например константой. Например, поставщик store сообщений может использовать путь к папке в качестве ключа уведомления. 
  
Ключ уведомления должен работать в нескольких процессах. 
  
Требования к области для ключа уведомлений похожи на требования к долгосрочному идентификатору записи. Однако, в отличие от идентификатора записи, ключ уведомления должен быть сравним двоичным. Как правило, ключ уведомления включает значение **GUID,** определенное поставщиком услуг, за которым следуют другие сведения, характерные для конкретного поставщика, уникальные для объекта. 
  
О том, как использовать структуру **NOTIFKEY** для управления подключениями между тактами и объектами, которые создают уведомления, см. в подстроке "Поддержка уведомлений [о событиях".](supporting-event-notification.md)
