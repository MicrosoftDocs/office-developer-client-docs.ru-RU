# <a name="registering-service-provider-unique-identifiers"></a>Регистрация уникальных идентификаторов поставщика услуг

  
  
**Относится к**: Outlook 2013 | Outlook 2016 
  
Поставщики адресной книги, хранения сообщений и транспорта используют уникальный идентификатор [MAPIUID](mapiuid.md) для регистрации в объектах-службах различных типов. **MAPIUID** — это 16-byte-идентификатор, содержащий GUID. **MapIUID можно** создать с помощью следующей процедуры: 
  
1. Определите константы.
    
2. Вызов средства Visual Studio *GUID**. 
    
Например, поставщик адресной книги может включать следующую константу в файле заголовок для определения **MAPIUID:**
  
 `#define AB_UID_PROVIDER { 0Xe3, 0x3c, 0x67, 0xa0, \ 0xc8, 0x1f, 0x11, 0xce, \ 0xb2, 0xe4, 0x0, 0xaa, \ 0x0, 0x51, 0xe, 0x3b }`
  
 **Регистрация MAPIUID, если ваш поставщик является адресной книгой или поставщиком магазина сообщений**
  
1. Вызов [IMAPISupport::SetProviderUID](imapisupport-setprovideruid.md).
    
2. Зарегистрируйте **MAPIUID** для каждого создаемого объекта входа и включите **этот MAPIUID** в первые 16 bytes **ab-члена** каждого создадаемого идентификатора записи. MAPI использует **структуры MAPIUID** для связи объектов с поставщиками услуг. Когда клиент вызывает метод [IMAPISession::OpenEntry](imapisession-openentry.md) для открытия объекта, MAPI проверяет часть **MAPIUID** идентификатора записи, соотнося ее с зарегистрированным **MAPIUID,** чтобы определить, какой объект входа должен получить открытый запрос.
    
3. Если ваш поставщик является транспортом, зарегистрируйте одну или несколько структур **MAPIUID,** когда MAPI вызывает метод **IXPLogon::AddressTypes.** MAPI использует структуры **MAPIUID,** зарегистрированные поставщиками транспорта, для назначения ответственности за доставку сообщений. 
    
Хотя поставщики услуг обычно регистрируют один **MAPIUID,** можно зарегистрировать несколько **структур MAPIUID.** Если ваша адресная книга или поставщик службы хранения сообщений поддерживает несколько объектов для логотипа, например, разрешив пользователю добавлять в свой профиль несколько экземпляров поставщика, может потребоваться реализовать разные **mapIUID** для каждого объекта для логотипа. Существует несколько других причин для поддержки нескольких **MAPIUID:**
  
- Необходимо поддерживать несколько версий поставщика, а идентификаторы записей должны представлять соответствующую версию. Назначьте **разные MAPIUID** для каждой версии. 
    
- Необходимо различать типы поддерживаемых объектов. Например, поставщику адресной книги может потребоваться зарегистрировать **один MAPIUID** для использования в идентификаторах записей объектов пользователей обмена сообщениями, а другой **MAPIUID** — для списков рассылки. 
    
Если одновременно активно несколько объектов для логотипа, имеет смысл использовать уникальные структуры **MAPIUID** для каждого из них. Это повышает точность, с которой MAPI соответствует идентификаторам записей поставщикам услуг и экономит часть работы. Если у каждого объекта для логотипа есть свой уникальный идентификатор, MAPI может гарантировать, что любой запрос, который он перенаправляет в объект для логотипа, может обрабатываться этим объектом. Когда объекты для логотипа имеют обную структуру **MAPIUID,** MAPI маршрутит запрос на первый объект для логотипа, который определен **MAPIUID.** Если один из объектов входа получает запрос, который не может обработать, так как он не обрабатывает идентификатор записи, перед возвратом ошибки передайте его следующему объекту входа.
