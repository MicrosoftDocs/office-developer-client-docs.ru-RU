# <a name="mapi-message-classes"></a><span data-ttu-id="23924-103">Классы сообщений MAPI</span><span class="sxs-lookup"><span data-stu-id="23924-103">MAPI Message Classes</span></span>

  
  
<span data-ttu-id="23924-104">**Относится к**: Outlook 2013 | Outlook 2016</span><span class="sxs-lookup"><span data-stu-id="23924-104">**Applies to**: Outlook 2013 | Outlook 2016</span></span> 
  
<span data-ttu-id="23924-105">Каждое сообщение имеет свойство класс сообщения, **PR_MESSAGE_CLASS** ([PidTagMessageClass](pidtagmessageclass-canonical-property.md)), который определяет тип, назначение или содержимое сообщения.</span><span class="sxs-lookup"><span data-stu-id="23924-105">Every message has a message class property, **PR_MESSAGE_CLASS** ([PidTagMessageClass](pidtagmessageclass-canonical-property.md)), which identifies the type, purpose, or content of the message.</span></span> <span data-ttu-id="23924-106">**PR_MESSAGE_CLASS** свойство является обязательным для всех новых сообщений.</span><span class="sxs-lookup"><span data-stu-id="23924-106">**PR_MESSAGE_CLASS** is a required property on all new messages.</span></span> <span data-ttu-id="23924-107">Класс сообщения определяет формы, в которой используется для представления сообщение пользователю и папка для размещения входящих сообщений.</span><span class="sxs-lookup"><span data-stu-id="23924-107">A message's class determines the form that is used to present the message to the user and the folder for placing incoming messages.</span></span> 
  
<span data-ttu-id="23924-108">Классы сообщений являются строками с учетом регистра символов, которые содержат символы ASCII 32 до 127, которые отделяются периодов, но не может заканчиваться точкой.</span><span class="sxs-lookup"><span data-stu-id="23924-108">Message classes are case-sensitive character strings that contain ASCII characters 32 through 127 and are delimited by periods, but they cannot end with a period.</span></span> <span data-ttu-id="23924-109">Каждая строка представляет уровень подклассы и нет ограничений на количество допустимых уровней.</span><span class="sxs-lookup"><span data-stu-id="23924-109">Each string represents a level of subclassing, and there is no limit to the number of levels allowed.</span></span> 
  
<span data-ttu-id="23924-110">Например, класса большинство сообщений, что клиентские приложения отправки и получения попадают в сообщение **IPM** широкий категорию, которая описывает все сообщения электронной почты — это (то есть, сообщения, предназначенные для чтения человеческого пользователем, а не программными средствами с помощью компьютер).</span><span class="sxs-lookup"><span data-stu-id="23924-110">For example, most messages that client applications send and receive fall into the **IPM** message class, a broad category that describes all interpersonal messages (that is, messages that are meant to be read by a human user, rather than programmatically by a computer).</span></span> <span data-ttu-id="23924-111">Сообщение об IPM точнее описывают поставщиков хранилища сообщений, создание подкласса **IPM** .</span><span class="sxs-lookup"><span data-stu-id="23924-111">Message store providers more precisely describe an IPM message by creating an **IPM** subclass.</span></span> <span data-ttu-id="23924-112">Подкласс **IPM** наследует свойства класса сообщений **IPM** .</span><span class="sxs-lookup"><span data-stu-id="23924-112">The **IPM** subclass inherits the properties of the **IPM** message class.</span></span> <span data-ttu-id="23924-113">Подклассов класса **IPM** называются путем объединения других знаков на идентификатор IPM, такие как **IPM. Примечание** для описания сообщение заметки и **IPM. Обратитесь в** для описания контактов сообщения.</span><span class="sxs-lookup"><span data-stu-id="23924-113">Subclasses of the **IPM** class are named by concatenating other character strings onto the IPM identifier, such as **IPM.Note** to describe a note message and **IPM.Contact** to describe a contact message.</span></span> 
  
<span data-ttu-id="23924-114">Для отображения и управления IPM сообщений, клиенты могут использовать стандартной формы, который предоставляет MAPI.</span><span class="sxs-lookup"><span data-stu-id="23924-114">To handle the display and management of IPM messages, clients can use a standard form that MAPI provides.</span></span> <span data-ttu-id="23924-115">Для отображения и управления новые классы сообщений, как разработчик приложения клиента имеется два варианта:</span><span class="sxs-lookup"><span data-stu-id="23924-115">To handle the display and management of new message classes, you as a client application developer have two options:</span></span>
  
1. <span data-ttu-id="23924-116">Можно создать новую форму с помощью набор интерфейсов определенные MAPI формы, которые можно использовать стандартный клиента.</span><span class="sxs-lookup"><span data-stu-id="23924-116">You can create a new form by using the set of MAPI-defined form interfaces that a standard client can use.</span></span>
    
2. <span data-ttu-id="23924-117">Можно написать собственного клиента при помощи завершено, автономного приложения.</span><span class="sxs-lookup"><span data-stu-id="23924-117">You can write your own client by implementing a complete, standalone application.</span></span> 
    
<span data-ttu-id="23924-118">Хотя клиенты следует установить свойство **PR_MESSAGE_CLASS** для всех исходящих сообщений на подкласс класса **IPM** или **производственных Издержек**, поставщик хранения сообщений имеет ultimate ответственность за установка его.</span><span class="sxs-lookup"><span data-stu-id="23924-118">Although clients should set the **PR_MESSAGE_CLASS** property for every outgoing message to a subclass of either **IPM** or **IPC**, the message store provider has the ultimate responsibility for setting it.</span></span> <span data-ttu-id="23924-119">Таким образом Если клиент отправляет сообщение без установки его класс сообщения, поставщик хранения сообщений устанавливает соответствующее значение по умолчанию для соответствующего типа клиента.</span><span class="sxs-lookup"><span data-stu-id="23924-119">Therefore, if a client sends a message without setting its message class, the message store provider sets it to the appropriate default value for the appropriate type of client.</span></span> <span data-ttu-id="23924-120">Класс сообщения по умолчанию для электронной почты — это клиенты обмена сообщениями является **IPM**; Класс сообщения по умолчанию для клиентов, связи между процессами — **производственных Издержек**.</span><span class="sxs-lookup"><span data-stu-id="23924-120">The default message class for interpersonal messaging clients is **IPM**; the default message class for interprocess communication clients is **IPC**.</span></span> 
  
<span data-ttu-id="23924-121">Классы сообщений имеют ограничение длины 255 символов.</span><span class="sxs-lookup"><span data-stu-id="23924-121">Message classes have a length restriction of 255 characters.</span></span> <span data-ttu-id="23924-122">Тем не менее классы сообщений не должна превышать 127 знаков для поддержки классы сообщений, используемые в отчетах.</span><span class="sxs-lookup"><span data-stu-id="23924-122">However, message classes should not exceed 127 characters to support the message classes used in reports.</span></span> <span data-ttu-id="23924-123">Классы сообщений отчета основаны на класс исходное сообщение с двумя дополнения: префикс и суффикс.</span><span class="sxs-lookup"><span data-stu-id="23924-123">Report message classes are based on the class of the original message, with two additions: a prefix and a suffix.</span></span> <span data-ttu-id="23924-124">Префикс ОТЧЕТОВ указывает, что сообщение является подчиненным и суффикс указывает тип отчета: аварийное Восстановление (отчет о), отчет о Недоставке (отчет о недоставке), IPNRN (чтение отчета) или IPNNRN (nonread отчет).</span><span class="sxs-lookup"><span data-stu-id="23924-124">The prefix REPORT indicates that the message is a report, and the suffix indicates the type of report: DR (delivery report), NDR (nondelivery report), IPNRN (read report), or IPNNRN (nonread report).</span></span> <span data-ttu-id="23924-125">Обратите внимание на то, что эти ограничения на длину, приводятся в символов; для платформ, использующих набор двухбайтовых знаков количество фактических байтов может быть выше.</span><span class="sxs-lookup"><span data-stu-id="23924-125">Note that these length restrictions are given in characters; on platforms that use a double-byte character set, the actual byte count might be higher.</span></span> 
  
<span data-ttu-id="23924-126">Поставщики хранилища сообщений должен возвращать MAPI_E_INVALID_PARAMETER из их реализации метода [IMAPIProp::SetProps](imapiprop-setprops.md) при попытке клиента назначение строку, которая превышает допустимое для их класс сообщения.</span><span class="sxs-lookup"><span data-stu-id="23924-126">Message store providers should return MAPI_E_INVALID_PARAMETER from their [IMAPIProp::SetProps](imapiprop-setprops.md) method implementations when a client attempts to assign a string that exceeds the allowable limit for their message class.</span></span>
