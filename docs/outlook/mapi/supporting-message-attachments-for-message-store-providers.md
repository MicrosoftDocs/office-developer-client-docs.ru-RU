<span data-ttu-id="19781-104">**Относится к**: Outlook 2013 | Outlook 2016</span><span class="sxs-lookup"><span data-stu-id="19781-104">**Applies to**: Outlook 2013 | Outlook 2016</span></span> 
  
<span data-ttu-id="19781-105">Поставщику вашего магазина сообщений не нужно поддерживать вложения сообщений.</span><span class="sxs-lookup"><span data-stu-id="19781-105">Your message store provider does not need to support message attachments.</span></span> <span data-ttu-id="19781-106">Однако многие клиентские приложения могут добавлять вложения в сообщения.</span><span class="sxs-lookup"><span data-stu-id="19781-106">However, many client applications expect to be able to add attachments to messages.</span></span> <span data-ttu-id="19781-107">Если хранилище сообщений будет использоваться для создания или хранения IPM. Обратите внимание на сообщения, после чего следует поддерживать вложения сообщений.</span><span class="sxs-lookup"><span data-stu-id="19781-107">If your message store will be used to create or store IPM.Note messages, then you should support message attachments.</span></span> <span data-ttu-id="19781-108">Поставщики хранения сообщений по умолчанию также должны поддерживать вложения сообщений.</span><span class="sxs-lookup"><span data-stu-id="19781-108">Default message store providers should also support message attachments.</span></span> <span data-ttu-id="19781-109">Дополнительные сведения см. в классах сообщений [MAPI](mapi-message-classes.md)и [хранилищах сообщений по умолчанию.](default-message-stores.md)</span><span class="sxs-lookup"><span data-stu-id="19781-109">For more information, see [MAPI Message Classes](mapi-message-classes.md), and [Default Message Stores](default-message-stores.md).</span></span>
  
<span data-ttu-id="19781-110">MAPI поддерживает пять типов вложений: вложенные файлы, вложения данных, вложения сообщений, вложения объектов OLE и ссылки.</span><span class="sxs-lookup"><span data-stu-id="19781-110">There are five types of attachments that MAPI supports: file attachments, data attachments, message attachments, OLE object attachments, and links.</span></span> <span data-ttu-id="19781-111">Требования для поддержки каждого типа отличаются.</span><span class="sxs-lookup"><span data-stu-id="19781-111">The requirements for supporting each type are different.</span></span> <span data-ttu-id="19781-112">Клиенты различают два типа вложений с помощью свойства **PR_ATTACH_METHOD** ([PidTagAttachMethod)](pidtagattachmethod-canonical-property.md)объектов вложений.</span><span class="sxs-lookup"><span data-stu-id="19781-112">Clients differentiate between the two types of attachments by means of the **PR_ATTACH_METHOD** ([PidTagAttachMethod](pidtagattachmethod-canonical-property.md)) property on attachment objects.</span></span>
  
<span data-ttu-id="19781-113">Поддержка вложений означает реализацию [интерфейса IAttach : IMAPIProp.](iattachimapiprop.md)</span><span class="sxs-lookup"><span data-stu-id="19781-113">Supporting attachments means implementing the [IAttach : IMAPIProp](iattachimapiprop.md) interface.</span></span> <span data-ttu-id="19781-114">Интерфейс **IAttach** не имеет собственных методов; он имеет только методы, наследуемые от [интерфейса IMAPIProp.](imapipropiunknown.md)</span><span class="sxs-lookup"><span data-stu-id="19781-114">The **IAttach** interface has no methods of its own; it has only methods that are inherited from the [IMAPIProp](imapipropiunknown.md) interface.</span></span> <span data-ttu-id="19781-115">Так как поставщик хранилище сообщений уже должен реализовать свойства для объектов сообщений, это значительно упрощает задачу поддержки вложений.</span><span class="sxs-lookup"><span data-stu-id="19781-115">Because your message store provider must already implement properties for message objects, this greatly simplifies the task of supporting attachments.</span></span> <span data-ttu-id="19781-116">Реализация **IAttach по** сути означает предоставление клиентам доступа к таблице свойств для определенных вложений в сообщениях.</span><span class="sxs-lookup"><span data-stu-id="19781-116">Implementing **IAttach** basically means providing a way for clients to access a table of properties for particular attachments on messages.</span></span> 
  
<span data-ttu-id="19781-117">Вложения данных — это просто вложения, для которых содержимое вложения хранится непосредственно в свойстве **PR_ATTACH_DATA_BIN** [(PidTagAttachDataBinary).](pidtagattachdatabinary-canonical-property.md)</span><span class="sxs-lookup"><span data-stu-id="19781-117">Data attachments are simply attachments for which the contents of the attachment are stored directly in an attachment's **PR_ATTACH_DATA_BIN** ([PidTagAttachDataBinary](pidtagattachdatabinary-canonical-property.md)) property.</span></span> <span data-ttu-id="19781-118">Вложения данных в основном позволяют клиентам вложение файлов в сообщение, когда отправитель и получатель сообщения не имеют доступа к общему файловом серверу.</span><span class="sxs-lookup"><span data-stu-id="19781-118">Data attachments exist primarily to allow clients to attach files to a message when the sender and the recipient of the message do not have access to a common file server.</span></span> <span data-ttu-id="19781-119">Дополнительные сведения см. в **PR_ATTACH_METHOD** ([PidTagAttachMethod).](pidtagattachmethod-canonical-property.md)</span><span class="sxs-lookup"><span data-stu-id="19781-119">For more information, see the **PR_ATTACH_METHOD** ([PidTagAttachMethod](pidtagattachmethod-canonical-property.md)) property.</span></span>
  
<span data-ttu-id="19781-120">Вложения сообщений — это вложения, для которых вложенный объект является другим [объектом IMessage : MAPIProp.](imessageimapiprop.md)</span><span class="sxs-lookup"><span data-stu-id="19781-120">Message attachments are attachments for which the attachment subobject is another [IMessage : MAPIProp](imessageimapiprop.md) object.</span></span> <span data-ttu-id="19781-121">Так как поставщики хранимых сообщений уже поддерживают **интерфейс IMessage,** поддержка вложений сообщений не является сложной задачей.</span><span class="sxs-lookup"><span data-stu-id="19781-121">Because message store providers already support the **IMessage** interface, supporting message attachments is not difficult.</span></span> 
  
<span data-ttu-id="19781-122">Поддержка вложений объектов OLE означает реализацию интерфейсов OLE **IStorage,** **IStream** и **IStreamDocfile.**</span><span class="sxs-lookup"><span data-stu-id="19781-122">Supporting OLE object attachments means implementing the OLE **IStorage**, **IStream**, and **IStreamDocfile** interfaces.</span></span> <span data-ttu-id="19781-123">Поставщик хранения сообщений должен иметь возможность преобразовывать данные объектов OLE, хранимые в сообщении, в активный объект OLE, когда клиент открывает объект.</span><span class="sxs-lookup"><span data-stu-id="19781-123">Your message store provider must be able to convert OLE object data stored in the message into an active OLE object when a client opens the object.</span></span> 
  
<span data-ttu-id="19781-124">Ссылки имеют два типа: ссылки на файлы и ссылки на другие сообщения.</span><span class="sxs-lookup"><span data-stu-id="19781-124">Links come in two types: links to files and links to other messages.</span></span> <span data-ttu-id="19781-125">Ссылки на файлы используют ATTACH_BY_REF_ONLY для свойства **PR_ATTACH_METHOD** вместе с **PR_ATTACH_PATHNAME** ([PidTagAttachPathname)](pidtagattachpathname-canonical-property.md)или **PR_ATTACH_LONG_PATHNAME** ([PidTagAttachLongPathname)](pidtagattachlongpathname-canonical-property.md)для указания расположения файла.</span><span class="sxs-lookup"><span data-stu-id="19781-125">Links to files use the ATTACH_BY_REF_ONLY value for the **PR_ATTACH_METHOD** property along with **PR_ATTACH_PATHNAME** ([PidTagAttachPathname](pidtagattachpathname-canonical-property.md)) or **PR_ATTACH_LONG_PATHNAME** ([PidTagAttachLongPathname](pidtagattachlongpathname-canonical-property.md)) to specify the location of a file.</span></span>
  
<span data-ttu-id="19781-126">Реализация ссылок на сообщения может зависеть от аспектов локальной системы обмена сообщениями, поэтому здесь нельзя полностью задокументировать.</span><span class="sxs-lookup"><span data-stu-id="19781-126">How one implements links to messages may depend on aspects of the local messaging system and, as such, cannot be fully documented here.</span></span> <span data-ttu-id="19781-127">Например, отправка ссылки на сообщение, которое хранится в серверном хранилище сообщений, обычно является лишь вопросом отправки идентификатора записи связанного сообщения, при этом у отправляющих и получателей есть доступ к этому серверу.</span><span class="sxs-lookup"><span data-stu-id="19781-127">For example, sending a link to a message that is stored on a server-based message store is typically just a matter of sending the entry identifier of the linked message, providing that both the sender and recipient have access to that server.</span></span> <span data-ttu-id="19781-128">Другие конфигурации системы обмена сообщениями предъявляют другие требования и трудности при реализации ссылок на сообщения.</span><span class="sxs-lookup"><span data-stu-id="19781-128">Other messaging system configurations present other requirements and challenges for implementing links to messages.</span></span>
