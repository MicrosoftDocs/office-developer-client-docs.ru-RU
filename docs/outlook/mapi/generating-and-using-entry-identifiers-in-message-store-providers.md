# <a name="generating-and-using-entry-identifiers-in-message-store-providers"></a>Создание и использование идентификаторов записей в поставщиках хранилищ сообщений

**Область применения**: Outlook 2013 | Outlook 2016 
  
Когда в хранилище сообщений создается новая папка или сообщение, поставщик магазина сообщений должен назначить этому объекту идентификатор входа, чтобы клиентские приложения могли ссылаться на него. Поставщики магазинов сообщений могут либо повторно использовать несуществуют долгосрочные идентификаторы записей удаленных объектов, либо создавать новые идентификаторы. Для поставщиков магазинов сообщений не существует требований, в том или ином случае; однако, если это возможно, поставщик магазина сообщений всегда должен создавать новые долгосрочные идентификаторы записей для новых объектов, а не повторно использовать старые. Если объекты, на которые они ссылаются, можно повторно использовать краткосрочные идентификаторы записей.
  
Причина этого удаления заключается в том, что клиенты могут кэшаторы идентификаторов записей, иногда в течение длительных периодов времени. Если это произойдет, и поставщик магазина сообщений повторно будет использовать идентификаторы записей, идентификатор записи может ссылаться на другой объект, когда клиент открывает идентификатор записи, чем при первом получении идентификатора записи. Если поставщик хранения сообщений не использует идентификаторы записей повторно или, по крайней мере, использует схему генерации идентификатора записи, которая не повторяется в течение очень долгого времени, эта проблема не может возникнуть.
  
Кроме того, поставщики магазинов сообщений должны пытаться сохранить идентификаторы записей для папок и сообщений при перемещении в хранилище сообщений. Если поставщик хранения сообщений может это сделать, ссылки на объекты в магазине не станут недействительными при перемещении объекта в другое расположение в магазине.
