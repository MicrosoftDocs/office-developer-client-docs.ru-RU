# <a name="generating-and-using-entry-identifiers-in-message-store-providers"></a>Создание и использование идентификаторов записей в поставщиках хранилищ сообщений

**Относится к**: Outlook 2013 | Outlook 2016 
  
Когда в хранилище сообщений создается новая папка или сообщение, поставщик хранилища сообщений должен назначить этому объекту идентификатор записи, чтобы клиентские приложения могли ссылаться на него. Поставщики хранилища сообщений могут повторно использовать устаревшие долгосрочные идентификаторы записей удаленных объектов или создавать новые идентификаторы. Нет необходимости одним способом или другой для поставщиков хранилищ сообщений; Однако если это возможно, поставщик хранилища сообщений всегда должен создавать новые долгосрочные идентификаторы записей для новых объектов, а не использовать старые. Можно использовать краткосрочные идентификаторы записей, когда объекты, на которые они ссылаются, будут удалены.
  
Причина этого удаления заключается в том, что клиенты могут кэшировать идентификаторы записей, иногда в течение длительных периодов времени. Если это произойдет, а поставщик хранилища сообщений повторно использует идентификаторы записей, то идентификатор записи может ссылаться на другой объект, когда клиент открывает идентификатор записи, а не при первом получении идентификатора записи. Если поставщик хранилища сообщений не может повторно использовать идентификаторы записей, или по крайней мере, использует схему создания идентификатора записи, которая не повторяется в течение очень долгого времени, эта проблема не может возникнуть.
  
Аналогично, поставщики хранилища сообщений должны попытаться сохранять идентификаторы записей для папок и сообщений при их перемещении в хранилище сообщений. Если поставщик хранилища сообщений может сделать это, ссылки на объекты в хранилище станут недействительными при перемещении объекта в другое место в хранилище.
