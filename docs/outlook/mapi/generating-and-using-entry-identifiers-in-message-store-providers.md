# <a name="generating-and-using-entry-identifiers-in-message-store-providers"></a><span data-ttu-id="d1faf-103">Создание и использование идентификаторов записей в поставщиках хранилищ сообщений</span><span class="sxs-lookup"><span data-stu-id="d1faf-103">Generating and using entry identifiers in message store providers</span></span>

<span data-ttu-id="d1faf-104">**Область применения**: Outlook 2013 | Outlook 2016</span><span class="sxs-lookup"><span data-stu-id="d1faf-104">**Applies to**: Outlook 2013 | Outlook 2016</span></span> 
  
<span data-ttu-id="d1faf-105">Когда в хранилище сообщений создается новая папка или сообщение, поставщик хранилища сообщений должен назначить этому объекту идентификатор записи, чтобы клиентские приложения могли ссылаться на него.</span><span class="sxs-lookup"><span data-stu-id="d1faf-105">When a new folder or message is created in a message store, the message store provider has to assign that object an entry identifier so that client applications can refer to it.</span></span> <span data-ttu-id="d1faf-106">Поставщики хранилища сообщений могут повторно использовать устаревшие долгосрочные идентификаторы записей удаленных объектов или создавать новые идентификаторы.</span><span class="sxs-lookup"><span data-stu-id="d1faf-106">Message store providers can either reuse the defunct long-term entry identifiers of deleted objects or create new identifiers.</span></span> <span data-ttu-id="d1faf-107">Нет необходимости одним способом или другой для поставщиков хранилищ сообщений; Однако если это возможно, поставщик хранилища сообщений всегда должен создавать новые долгосрочные идентификаторы записей для новых объектов, а не использовать старые.</span><span class="sxs-lookup"><span data-stu-id="d1faf-107">There is no requirement one way or the other for message store providers; however, if it is feasible, a message store provider should always generate new long-term entry identifiers for new objects rather than reusing old ones.</span></span> <span data-ttu-id="d1faf-108">Можно использовать краткосрочные идентификаторы записей, когда объекты, на которые они ссылаются, будут удалены.</span><span class="sxs-lookup"><span data-stu-id="d1faf-108">It is fine to reuse short-term entry identifiers when the objects they refer to are deleted.</span></span>
  
<span data-ttu-id="d1faf-109">Причина этого удаления заключается в том, что клиенты могут кэшировать идентификаторы записей, иногда в течение длительных периодов времени.</span><span class="sxs-lookup"><span data-stu-id="d1faf-109">The reason for this deletion is that clients can cache entry identifiers, sometimes for long periods of time.</span></span> <span data-ttu-id="d1faf-110">Если это произойдет, а поставщик хранилища сообщений повторно использует идентификаторы записей, то идентификатор записи может ссылаться на другой объект, когда клиент открывает идентификатор записи, а не при первом получении идентификатора записи.</span><span class="sxs-lookup"><span data-stu-id="d1faf-110">If that happens and the message store provider does reuse entry identifiers, it is possible for the entry identifier to refer to a different object when the client opens the entry identifier than when it first obtained the entry identifier.</span></span> <span data-ttu-id="d1faf-111">Если поставщик хранилища сообщений не может повторно использовать идентификаторы записей, или по крайней мере, использует схему создания идентификатора записи, которая не повторяется в течение очень долгого времени, эта проблема не может возникнуть.</span><span class="sxs-lookup"><span data-stu-id="d1faf-111">If the message store provider does not reuse entry identifiers — or at least uses an entry identifier generation scheme that does not repeat for a very long time — this problem cannot occur.</span></span>
  
<span data-ttu-id="d1faf-112">Аналогично, поставщики хранилища сообщений должны попытаться сохранять идентификаторы записей для папок и сообщений при их перемещении в хранилище сообщений.</span><span class="sxs-lookup"><span data-stu-id="d1faf-112">Similarly, message store providers should attempt to preserve entry identifiers for folders and messages when they are moved in the message store.</span></span> <span data-ttu-id="d1faf-113">Если поставщик хранилища сообщений может сделать это, ссылки на объекты в хранилище станут недействительными при перемещении объекта в другое место в хранилище.</span><span class="sxs-lookup"><span data-stu-id="d1faf-113">If the message store provider can do that, references to objects in the store will not become invalid when the object is moved to a different location in the store.</span></span>
