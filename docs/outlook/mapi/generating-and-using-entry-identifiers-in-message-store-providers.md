# <a name="generating-and-using-entry-identifiers-in-message-store-providers"></a>Создание и использование идентификаторов записей в поставщиках хранилищ сообщений

**Относится к**: Outlook 2013 | Outlook 2016 
  
Когда в хранилище сообщений создается новая папка или сообщение, поставщику store сообщений необходимо назначить этому объекту идентификатор записи, чтобы клиентские приложения могли ссылаться на него. Поставщики хранимых сообщений могут либо повторно использовать несуществуют долгосрочные идентификаторы записей удаленных объектов, либо создавать новые идентификаторы. Для поставщиков store сообщений нет требований; Однако, если это возможно, поставщик хранилище сообщений всегда должен создавать новые долгосрочные идентификаторы записей для новых объектов, а не повторно использовать старые. Можно повторно использовать краткосрочные идентификаторы записей при удалении объектов, на которые они ссылаются.
  
Причиной такого удаления является то, что клиенты могут кэшаторить идентификаторы записей, иногда в течение длительного времени. Если это произойдет, и поставщик store сообщений повторно будет использовать идентификаторы записей, идентификатор записи может ссылаться на другой объект, когда клиент открывает идентификатор записи, чем при первом получении идентификатора записи. Если поставщик хранения сообщений не использует идентификаторы записей повторно или, по крайней мере, использует схему генерации идентификаторов записей, которая не повторяется в течение очень долгого времени, эта проблема не может возникнуть.
  
Аналогичным образом, поставщики store сообщений должны пытаться сохранить идентификаторы записей для папок и сообщений при их перемещении в хранилище сообщений. Если поставщик хранения сообщений может сделать это, ссылки на объекты в хранилище не станут недопустимыми при перемещении объекта в другое расположение в хранилище.
