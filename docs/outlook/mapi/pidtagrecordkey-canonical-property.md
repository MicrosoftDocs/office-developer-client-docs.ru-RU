# <a name="pidtagrecordkey-canonical-property"></a>Каноническое свойство PidTagRecordKey

  
  
**Область применения**: Outlook 2013 | Outlook 2016 
  
Содержит уникальный двоичный код, сравнимый с двоичным кодом для определенного объекта.
  
|||
|:-----|:-----|
|Связанные свойства:  <br/> |ПР_РЕКОРД_КЭЙ  <br/> |
|Идентификатор:  <br/> |0x0FF9  <br/> |
|Тип данных:  <br/> |PT_BINARY  <br/> |
|Область:  <br/> |Свойства идентификатора  <br/> |
   
## <a name="remarks"></a>Примечания

Это свойство облегчает поиск ссылок на объект, например поиск строки в таблице содержимого. Это свойство невозможно использовать для открытия объекта; Используйте идентификатор записи для этой цели.
  
Это свойство должно однозначно определять вложенный в сообщение вложенный вложенный объект. Этот идентификатор является единственной гарантией вложения, которая гарантированно остается неизменной после закрытия и повторного открытия сообщения. Чтобы гарантировать эту гарантию, поставщик хранилища должен сохранить это свойство в сеансах.
  
Для папок это свойство содержит ключ, используемый в таблице иерархия папок. Как правило, это то же значение, которое предоставляется свойством **пр_ентрид** ([PidTagEntryId](pidtagentryid-canonical-property.md)).
  
Для хранилищ сообщений это свойство идентично свойству **пр_сторе_рекорд_кэй** ([PidTagStoreRecordKey](pidtagstorerecordkey-canonical-property.md)).
  
В объекте хранилища сообщений это свойство должно быть уникальным для всех поставщиков хранилищ. Один из способов сделать это — объединить значение свойства **пр_мдб_провидер** ([PidTagStoreProvider](pidtagstoreprovider-canonical-property.md)) для хранилища (уникальное для этого типа поставщика) с использованием структуры [GUID](guid.md) или другого значения, уникального для конкретного хранилища сообщений. 
  
Это свойство всегда доступно с помощью метода [IMAPIProp::](imapiprop-getprops.md) -Props, следующего за первым вызовом метода [IMAPIProp:: SaveChanges](imapiprop-savechanges.md) . Некоторые поставщики могут сделать его доступным сразу же после создания экземпляра. 
  
Клиент или поставщик услуг может сравнить значения этого свойства с помощью мемкмп. Это невозможно сделать для значений идентификатора записи. Тем не менее, это свойство обязательно должно быть уникальным в том же хранилище сообщений или в контейнере адресной книги; два объекта из разных контейнеров могут иметь одно и то же значение этого свойства.
  
Одно из различий между ключами записи и поиска состоит в том, что ключ записи относится только к объекту, а ключ поиска может быть скопирован в другие объекты. Например, две копии объекта могут иметь одно и то же значение **пр_сеарч_кэй** ([PidTagSearchKey](pidtagsearchkey-canonical-property.md)), но для этого свойства должны быть заданы разные значения.
  
В следующей таблице приведены важные различия между **пр_ентрид**, **пр_сеарч_кэй** ([PidTagSearchKey](pidtagsearchkey-canonical-property.md)) и этим свойством. 
  
|**Характеристика**|**PR_ENTRYID**|**ПР_РЕКОРД_КЭЙ**|**PR_SEARCH_KEY**|
|:-----|:-----|:-----|:-----|
|Обязательный атрибут для объектов вложений  <br/> |Нет  <br/> |Да  <br/> |Нет  <br/> |
|Обязательный атрибут для объектов Folder  <br/> |Да  <br/> |Да  <br/> |Нет  <br/> |
|Обязательный для объектов хранилища сообщений  <br/> |Да  <br/> |Да  <br/> |Нет  <br/> |
|Обязательные для объектов Status  <br/> |Да  <br/> |Нет  <br/> |Нет  <br/> |
|Создаваемое клиентом  <br/> |Нет  <br/> |Нет  <br/> |Да  <br/> |
|Доступно до вызова **SaveChanges** <br/> |Возможно  <br/> |Возможно  <br/> |Сообщения Да, другие могут быть  <br/> |
|Изменено в операции копирования  <br/> |Да  <br/> |Да  <br/> |Нет  <br/> |
|Изменение клиентом после копирования  <br/> |Нет  <br/> |Нет  <br/> |Да  <br/> |
|Уникальный в пределах...  <br/> |Весь мир  <br/> |Экземпляр поставщика  <br/> |Весь мир  <br/> |
|Сравнение двоичных файлов (как и в случае с мемкмп)  <br/> |Нет — используйте **имаписуппорт:: метод compareentryids** <br/> |Да  <br/> |Да  <br/> |
