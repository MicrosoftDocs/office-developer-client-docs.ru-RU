# <a name="message-reception-model"></a>Модель приема сообщений

  
  
**Область применения**: Outlook 2013 | Outlook 2016 
  
Поставщик транспорта контролирует, должен ли шпалер MAPI опросить ее для входящих сообщений или выполняет ли он вызов обратно в пульпер MAPI по прибытии новой почты. Поставщик транспорта задает флаг SP_LOGON_POLL, когда он возвращается из [IXPProvider::TransportLogon](ixpprovider-transportlogon.md) для запроса опроса. В противном случае поставщик транспорта использует [IMAPISupport::SpoolerNotify](imapisupport-spoolernotify.md) при наличии входящих сообщений. Узнав о наличии входящих сообщений, пуллер MAPI открывает новое сообщение и просит поставщика транспорта хранить полученные свойства сообщения в сообщении. 
  
Этот процесс работает следующим образом:
  
1. Доступные сообщения указываются либо поставщиком транспорта по вызову **IMAPISupport::SpoolerNotify,** либо пулером MAPI, который вызывает [IXPLogon::P oll](ixplogon-poll.md).
    
2. Шпалер MAPI вызывает [IXPLogon::StartMessage](ixplogon-startmessage.md) для инициации процесса. 
    
3. Поставщик транспорта помещает справочные значения в расположении, на который ссылается **StartMessage.** Эти справочные значения позволяют поставщику транспорта и шпалеру MAPI отслеживать, какое сообщение обрабатывается при доставке нескольких сообщений.
    
4. Поставщик транспорта хранит данные сообщений в переданный [экземпляр IMessage : IMAPIProp.](imessageimapiprop.md) 
    
5. Поставщик транспорта вызывает [метод IMAPIProp::SaveChanges](imapiprop-savechanges.md) в **экземпляре IMessage** и возвращается из **StartMessage.**
    
6. Spooler MAPI вызывает [IXPLogon::TransportNotify,](ixplogon-transportnotify.md) если необходимо остановить доставку сообщений. 
    
> [!NOTE]
> Если поставщик транспорта должен доставить большое количество сообщений, а поставщик транспорта использует **IMAPISupport::SpoolerNotify** вместо **IXPLogon::P oll,** необходимо не вызывать **SpoolerNotify** слишком часто, чтобы не лишать других поставщиков транспорта времени ЦП. У шпалеров MAPI есть логика, чтобы предотвратить это, но в целом интервал между вызовами **SpoolerNotify** должен быть больше времени, которое требуется поставщику транспорта для обработки одного сообщения. > кроме того, шпалер MAPI может не обрабатывать входящие сообщения немедленно. Шпалер MAPI может попросить поставщика транспорта выполнить другие задачи перед получением входящих сообщений.
