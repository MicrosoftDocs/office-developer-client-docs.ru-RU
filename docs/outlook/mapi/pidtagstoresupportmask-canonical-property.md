# <a name="pidtagstoresupportmask-canonical-property"></a>Каноническое свойство PidTagStoreSupportMask

  
  
**Относится к**: Outlook 2013 | Outlook 2016 
  
Содержит битовуюmass флагов, которые клиентские приложения запрашивают для определения характеристик хранения сообщений. 
  
|||
|:-----|:-----|
|Связанные свойства:  <br/> |PR_STORE_SUPPORT_MASK  <br/> |
|Идентификатор:  <br/> |0x340D  <br/> |
|Тип данных:  <br/> |PT_LONG  <br/> |
|Область:  <br/> |Разное  <br/> |
   
## <a name="remarks"></a>Примечания

Это свойство раскрывает возможности хранения сообщений для клиентских приложений, которые планируют отправить ему сообщение. Флаги могут поддерживать решения клиента или другого магазина, такие как отправка PR_BODY **(** [PidTagBody)](pidtagbody-canonical-property.md)или только PR_RTF_COMPRESSED **(** [PidTagRtfCompressed).](pidtagrtfcompressed-canonical-property.md) Клиент никогда не должен **устанавливать** PR_STORE_SUPPORT_MASK; Попытка установить этот флаг возвращает MAPI_E_COMPUTED. 
  
Один или несколько из следующих флагов  можно установить для битовой PR_STORE_SUPPORT_MASK битов: 
  
STORE_ANSI_OK
  
> (131072, 0x00020000) Хранилище сообщений поддерживает свойства, содержащие символы ANSI (8-битные).
    
STORE_ATTACH_OK 
  
> (32, 0x00000020) Хранилище сообщений поддерживает вложения (OLE или non-OLE) для сообщений. 
    
STORE_CATEGORIZE_OK 
  
> (1024, 0x00000400) Хранилище сообщений поддерживает классификацию представлений таблиц. 
    
STORE_CREATE_OK 
  
> (16, 0x00000010) Хранилище сообщений поддерживает создание новых сообщений. 
    
STORE_ENTRYID_UNIQUE 
  
> (1, 0x00000001) Идентификаторы записей для объектов в хранилище сообщений уникальны, то есть никогда не повторно в течение всего жизненного года хранения. 
    
STORE_HTML_OK 
  
> (65536, 0x00010000) Хранилище сообщений поддерживает HTML-сообщения, хранимые в свойстве **PR_BODY_HTML** ([PidTagBodyHtml).](pidtagbodyhtml-canonical-property.md) Если в среде разработки используется MAPIDEFS. H-файл, который не включает STORE_HTML_OK, используйте значение 0x00010000. 
    
STORE_ITEMPROC
  
> (2097152, 0x00200000) В упакованном хранилище PST-данных указывает, что когда новое сообщение поступает в магазин, хранилище выполняет правила и обработку фильтра нежелательной почты отдельно для сообщения. Магазин вызывает [IMAPISupport::Notify](imapisupport-notify.md), setting **fnevNewMail** в структуре [NOTIFICATION,](notification.md) которая передается в качестве параметра, а затем передает сведения о новом сообщении клиенту прослушивания. Впоследствии, когда прослушивающий клиент получает уведомление, к сообщению не применяются правила. 
    
STORE_LOCALSTORE
  
> (524288, 0x00080000) Этот флаг зарезервирован и не должен использоваться.
    
STORE_MODIFY_OK 
  
> (8, 0x00000008) Хранилище сообщений поддерживает изменение существующих сообщений. 
    
STORE_MV_PROPS_OK 
  
> (512, 0x00000200) Хранилище сообщений поддерживает многооценные свойства, гарантирует стабильность порядка значений в многоценном свойстве в ходе операции сохранения и поддерживает возможность искомости многоценных свойств в таблицах. 
    
STORE_NOTIFY_OK 
  
> (256, 0x00000100) Хранилище сообщений поддерживает уведомления. 
    
STORE_OLE_OK 
  
> (64, 0x00000040) Хранилище сообщений поддерживает вложения OLE. Доступ к данным OLE можно получить через **интерфейс IStorage,** например через **свойство PR_ATTACH_DATA_OBJ** ([PidTagAttachDataObject).](pidtagattachdataobject-canonical-property.md) 
    
STORE_PUBLIC_FOLDERS 
  
> (16384, 0x00004000) Папки в этом хранилище являются общедоступными (несколькими пользователями), а не частными (возможно, с несколькими экземплярами, но не с несколькими пользователями). 
    
STORE_PUSHER_OK
  
> (8388608, 0x00800000) Обработщик протокола MAPI не будет обходить хранилище, и хранилище будет отвечать за то, чтобы индексировать сообщения с помощью уведомлений.
    
STORE_READONLY 
  
> (2, 0x00000002) Все интерфейсы для хранения сообщений имеют доступ только для чтения. 
    
STORE_RESTRICTION_OK 
  
> (4096, 0x00001000) Хранилище сообщений поддерживает ограничения. 
    
STORE_RTF_OK 
  
> (2048, 0x00000800) Хранилище сообщений поддерживает сообщения в формате RTF (обычно сжатые),  а само хранилище сохраняет PR_BODY и **PR_RTF_COMPRESSED** синхронизацию. 
    
STORE_RULES_OK
  
> (268435456, 0x10000000) Указывает, что правила должны храниться в этом PST-хранилище, даже если оно не является хранилищем по умолчанию. Если **STORE_RULES_OK** в сочетании с **NON_EMS_XP_SAVE,** правила могут запускаться в хранилищах, не используемых по умолчанию в PST-оболочке.
    
STORE_SEARCH_OK 
  
> (4, 0x00000004) Хранилище сообщений поддерживает папки результатов поиска. 
    
STORE_SORT_OK 
  
> (8192, 0x00002000) Хранилище сообщений поддерживает сортировку представлений таблиц. 
    
STORE_SUBMIT_OK 
  
> (128, 0x00000080) Хранилище сообщений поддерживает пометку сообщения для отправки. 
    
STORE_UNCOMPRESSED_RTF 
  
> (32768, 0x00008000) Хранилище сообщений поддерживает хранение сообщений RTF в несмеченной форме. Несхваченный поток RTF идентифицирован значением **dwMagicUncompressedRTF** в загоне потока. Значение **dwMagicUncompressedRTF** определяется в RTFLIB. H-файл. 
    
STORE_UNICODE_OK
  
> (262144, 0x00040000) Указывает, что хранилище сообщений поддерживает хранилище Юникод. Клиент можно проверить наличие флага, чтобы принять решение о запросе или сохранении сведений Юникода в хранилище. 
    
Версию сообщения в RTF всегда можно сохранить, даже если хранилище сообщений не имеет RTF-данных. Если STORE_RTF_OK для определенного магазина не заданной, клиент с версиями RTF должен сам вызывать функцию [RTFSync,](rtfsync.md) чтобы синхронизировать версии **PR_BODY** и **PR_RTF_COMPRESSED** для текстового содержимого. RTF всегда хранится в **PR_RTF_COMPRESSED** независимо от того, сжат ли он на самом деле.
