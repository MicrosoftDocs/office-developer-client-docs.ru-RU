# <a name="pidtagstoresupportmask-canonical-property"></a>Каноническое свойство PidTagStoreSupportMask

  
  
**Применимо к**: Outlook 2013 | Outlook 2016 
  
Содержит битовую маску флагов запроса клиентских приложений для определения характеристик хранилища сообщений. 
  
|||
|:-----|:-----|
|Связанные свойства:  <br/> |PR_STORE_SUPPORT_MASK  <br/> |
|Идентификатор:  <br/> |0x340D  <br/> |
|Тип данных:  <br/> |PT_LONG  <br/> |
|Область:  <br/> |Разное  <br/> |
   
## <a name="remarks"></a>Замечания

Это свойство, включающие описание возможностей хранилища сообщений на клиентские приложения, которые планируют отправить сообщение. Флаги может поддерживать решения с клиента или другого хранилища, например необходимость отправки **PR_BODY** ([PidTagBody](pidtagbody-canonical-property.md)) или только **PR_RTF_COMPRESSED** ([PidTagRtfCompressed](pidtagrtfcompressed-canonical-property.md)). Клиент никогда не следует устанавливать **PR_STORE_SUPPORT_MASK**; При попытке установить этот флаг возвращает MAPI_E_COMPUTED. 
  
Один или несколько из следующих флагов можно задать для битовой маски **PR_STORE_SUPPORT_MASK** : 
  
STORE_ANSI_OK
  
> (131072, 0x00020000) Хранилище сообщений поддерживает свойства, которые содержат символов ANSI (8 бит).
    
STORE_ATTACH_OK 
  
> (32, 0x00000020) Хранилище сообщений поддерживает (OLE или не OLE) вложений в сообщения. 
    
STORE_CATEGORIZE_OK 
  
> (1024, 0x00000400) Хранилище сообщений поддерживает по категориям представления таблицы. 
    
STORE_CREATE_OK 
  
> (16, 0x00000010) Хранилище сообщений поддерживает создание новых сообщений. 
    
STORE_ENTRYID_UNIQUE 
  
> (1, 0x00000001) Идентификаторы записей для объектов в хранилище сообщений являются уникальными, то есть, никогда не повторно использовать на протяжении хранилища. 
    
STORE_HTML_OK 
  
> (65536, 0x00010000) Хранилище сообщений поддерживает HTML-сообщений, хранящиеся в свойстве **PR_BODY_HTML** ([PidTagBodyHtml](pidtagbodyhtml-canonical-property.md)). Если в среде разработки используются MAPIDEFS. H файл, который не включает STORE_HTML_OK, используйте значение 0x00010000. 
    
STORE_ITEMPROC
  
> (2097152, 0x00200000) В оболочке хранилище PST-файлов указывает, что при поступлении нового сообщения в хранилище, хранилище выполняет правил и фильтров нежелательной почты отдельно обработки в окне сообщения. Хранилище вызовы [IMAPISupport::Notify](imapisupport-notify.md)параметр **fnevNewMail** в структуре [УВЕДОМЛЕНИЙ](notification.md) , который передается как параметр, а затем передает сведения о новых сообщений для прослушивания клиента. Следовательно когда прослушивания клиент получает уведомление, не будет обрабатывать правил для сообщения. 
    
STORE_LOCALSTORE
  
> (524288, 0x00080000) Этот флаг зарезервирован и не должен использоваться.
    
STORE_MODIFY_OK 
  
> (8, 0x00000008) Хранилище сообщений поддерживает изменения существующего сообщения. 
    
STORE_MV_PROPS_OK 
  
> (512, 0x00000200) Хранилище сообщений поддерживает многозначные свойства, гарантирует стабильность значение заказ в свойством во всем сохранения операции и поддерживает создание многозначных свойств в таблицах. 
    
STORE_NOTIFY_OK 
  
> (256, 0x00000100) Хранилище сообщений поддерживает уведомления. 
    
STORE_OLE_OK 
  
> (64, 0x00000040) Хранилище сообщений поддерживает вложения OLE. Данные OLE может осуществляться через интерфейс **IStorage** , например, который доступен через свойство **PR_ATTACH_DATA_OBJ** ([PidTagAttachDataObject](pidtagattachdataobject-canonical-property.md)). 
    
STORE_PUBLIC_FOLDERS 
  
> (16384, 0x00004000) Папки в этом хранилище — общедоступный (многопользовательский) не закрытый (возможно нескольких экземпляров, но не нескольких пользователей). 
    
STORE_PUSHER_OK
  
> (8388608, 0x00800000) Обработчик протокола MAPI не будет выполнять обход контента хранилище и хранилище несет ответственность за передача все изменения через уведомления для компонента индексирования индексируются сообщений.
    
STORE_READONLY 
  
> (2, 0x00000002) Все интерфейсы для хранилища сообщений с уровнем доступа только для чтения. 
    
STORE_RESTRICTION_OK 
  
> (4096, 0x00001000) Хранилище сообщений поддерживает ограничения. 
    
STORE_RTF_OK 
  
> (2048, 0x00000800) Хранилище сообщений поддерживает форматированный текст (RTF) сообщения, обычно сжатием и само хранилище отслеживает **PR_BODY** и **PR_RTF_COMPRESSED** синхронизации. 
    
STORE_RULES_OK
  
> (268435456, 0x10000000) Указывает, что правила должны храниться в PST-хранилище даже в том случае, если это не хранилище по умолчанию. При использовании в сочетании с **NON_EMS_XP_SAVE** **STORE_RULES_OK** правил, могут работать на хранилищ PST-файлов в оболочку не по умолчанию.
    
STORE_SEARCH_OK 
  
> (4, 0x00000004) Хранилище сообщений поддерживает папки результатов поиска. 
    
STORE_SORT_OK 
  
> (8192, 0x00002000) Хранилище сообщений поддерживает сортировки представления таблицы. 
    
STORE_SUBMIT_OK 
  
> (128, 0x00000080) Хранилище сообщений поддерживает пометки сообщения для отправки. 
    
STORE_UNCOMPRESSED_RTF 
  
> (32768, 0x00008000) Хранилище сообщений поддерживает хранение сообщения в формате RTF в несжатую форму. Поток несжатую RTF определяется значение **dwMagicUncompressedRTF** в заголовке потока. Значение **dwMagicUncompressedRTF** определен в RTFLIB. H-файл. 
    
STORE_UNICODE_OK
  
> (262144, 0x00040000) Указывает, что хранилище сообщений поддерживает Юникод хранилища. Клиент может выполнять на наличие флагом, чтобы решить, следует ли для запроса или сохранения данных Юникод в хранилище. 
    
Более RTF версии сообщения могут всегда храниться, даже в том случае, если хранилище сообщений не поддерживающими формат RTF. Если бит STORE_RTF_OK не установлен для конкретного хранилища, обслуживание RTF версий клиента самого вызовите функцию [RTFSync](rtfsync.md) сохранение версий **PR_BODY** и **PR_RTF_COMPRESSED** , синхронизированы для текстовое содержимое. Формат RTF всегда сохраняется в **PR_RTF_COMPRESSED**ли фактически сжимается или нет.
