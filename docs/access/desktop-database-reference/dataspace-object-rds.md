# <a name="dataspace-object-rds"></a>Объект DataSpace (RDS)

**Область применения**: Access 2013, Office 2013

Создает клиентские прокси-объекты для пользовательских бизнес-объектов, расположенных на среднем уровне.

## <a name="remarks"></a>Примечания

Службе удаленных данных необходимы прокси-объекты бизнес-объектов, чтобы клиентский компонент может взаимодействовать с бизнес-объектами, расположенными на среднем уровне. Прокси упрощают упаковку, распаковку и транспортировку (маршализации) данных [recordset](recordset-object-ado.md) приложения через границы процесса или машины.

Служба удаленных данных использует **RDS. Метод CreateObject** объекта [DataSpace](createobject-method-rds.md) для создания прокси бизнес-объектов. Прокси-сервер бизнес-объектов динамически создается всякий раз, когда создается экземпляр его аналога бизнес-объекта среднего уровня. Служба удаленных данных поддерживает следующие протоколы: HTTP, HTTPS (HTTP Secure Sockets), DCOM и in-process (клиентские компоненты и бизнес-объект находятся на одном компьютере).

> [!NOTE]
> RDS ведет себя "без состояния" при **RDS. Объект DataSpace** использует протоколы HTTP или HTTPS. То есть все внутренние сведения о клиентской просьбе отбрасываются после того, как сервер возвращает ответ.

Хотя бизнес-объект, как представляется, существует в течение всего срока службы прокси-сервера бизнес-объекта, бизнес-объект фактически существует только до тех пор, пока не будет отправлен ответ на запрос. При выдаче запроса (то есть на бизнес-объекте вызывается метод), прокси открывает новое подключение к серверу, а сервер создает новый экземпляр бизнес-объекта. После ответа бизнес-объекта на запрос сервер уничтожает бизнес-объект и закрывает подключение.

Это поведение означает, что вы не можете передавать данные из одного запроса в другой с помощью свойства бизнес-объекта или переменной. Для сохраняемого состояния данных необходимо использовать другой механизм, например файл или аргумент метода.
