# <a name="the-significance-of-cursor-location"></a>The Significance of Cursor Location


**Применимо к**: Access 2013 | Office 2013

Каждые курсор использует временные ресурсы для хранения своих данных. Эти ресурсы может быть памяти, размер файла подкачки диска, временные файлы или даже временной базы данных. Курсор вызывает курсора *со стороны клиента* , эти ресурсы расположены на клиентском компьютере. Курсор вызывает курсора *на сервере* , эти ресурсы, находятся на сервере.

## <a name="client-side-cursors"></a>Записей на стороне клиента

В ADO, вызов для курсора со стороны клиента с помощью **adUseClient** **CursorLocationEnum**. С помощью курсор со стороны клиента не набора ключей сервер отправляет весь результирующий набор по сети на клиентский компьютер. На клиентском компьютере предоставляет и управляет временные ресурсы, необходимые для курсор и результирующий набор. Клиентское приложение может просматривать весь результирующий набор строк, которые требуется определить.

Статические и управляемые набором ключей записей на стороне клиента разместить значительные нагрузки на рабочей станции, если они содержат слишком много строк. Хотя все библиотеки курсора поддерживающего построение курсоры с тысяч строк, приложения, предназначенные для извлечения таких большие наборы строк может работать неправильно. Существуют исключения, конечно. Для некоторых приложений большого курсора со стороны клиента может быть вполне подходящим и производительности может не приведет к возникновению проблем.

Одно из очевидных преимуществ курсор на стороне клиента — быстрого получения ответа. После загрузки набора результатов на клиентский компьютер, перейдя по строкам очень быстрый интерфейс. Приложения обычно масштабируемое с записей на стороне клиента, так как требований к ресурсам курсора помещаются на каждом отдельном клиенте, а не на сервере.

## <a name="server-side-cursors"></a>Записей на стороне сервера

В ADO, вызываться для курсора на сервере с помощью **adUseServer** **CursorLocationEnum.** С курсором на сервере сервер управляет набора результатов с помощью ресурсам на компьютере сервера. Указатель на сервере возвращает только запрошенные данные по сети. Этот тип курсора в некоторых случаях можно обеспечить лучшую производительность, чем курсор со стороны клиента, особенно в случаях, когда большой сетевой трафик проблем.

Тем не менее, важно отметить, что курсор на сервере является — по крайней мере временная, занимающих ресурсы ценные ресурсы сервера для всех активных клиентов. Необходимо запланировать соответствующим образом Убедитесь, что серверного оборудования для управления всеми записей на стороне сервера, затребованы активных клиентов. Кроме того, курсора на сервере может быть медленным, поскольку предоставляет доступ только одну строку — нет курсора пакета.

Записей на стороне сервера полезны при вставке, обновлении или удалении записей. С помощью записей на стороне сервера может иметь несколько активных операторов на одном подключении.
